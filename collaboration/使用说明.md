# Collaboration 使用说明

这套目录不是项目说明文档，而是前后端协作工作区。

适合你们当前这种场景：

- 前端只改 `frontend/`
- 后端只改 `backend/`
- 前后端通过文档同步需求、契约、阻塞和验收

## 目录作用

### `architecture/`

放长期相对稳定的架构说明。

- 前端看 `architecture/frontend.md`
- 后端看 `architecture/backend.md`

用途：

- 新需求开始前，先快速确认各自架构边界
- 明确哪些文件/模块是本侧负责
- 避免对另一侧实现做错误假设

### `features/`

放“一个需求一份主文档”。

例如：

- `features/workflow-realtime-chat.md`

用途：

- 写清楚需求目标
- 写清楚当前现状
- 写清楚前后端各要改什么
- 写清楚契约、验收标准、风险点

规则：

- 一个跨前后端需求，只保留一份主文档
- 不要前端一份、后端一份分别写
- 最终结论尽量回填到这里

### `handoffs/`

放开发过程中的阻塞和补充请求。

- 前端提给后端：`handoffs/frontend-to-backend.md`
- 后端提给前端：`handoffs/backend-to-frontend.md`

用途：

- 某一侧开发时发现对方缺能力
- 某一侧需要对方补字段、补事件、补规则
- 某一侧需要对方确认展示或交互决策

规则：

- 新增问题时追加一条 `open`
- 解决后标记为 `closed`
- 如果已经形成明确结论，回填到对应 `features/` 文档

### `templates/`

放模板，方便复制新建。

- 新功能文档可以参考 `templates/feature-template.md`

## 推荐工作流

### 1. 新需求开始

先在 `features/` 新建一个文档，至少写清楚：

- 需求目标
- 当前行为
- 前端需要改什么
- 后端需要改什么
- 需要先确认的契约
- 验收标准

如果是跨前后端需求，不建议直接靠聊天开始写代码。

### 2. 各自开发

前端：

- 只改 `frontend/`
- 按 `features/` 文档实现前端部分

后端：

- 只改 `backend/`
- 按 `features/` 文档实现后端部分

### 3. 发现阻塞

如果前端发现后端缺字段、缺事件、缺接口：

- 不直接修改后端代码
- 在 `handoffs/frontend-to-backend.md` 追加一条记录

如果后端发现前端没有明确展示规则、交互规则：

- 不自己假设 UI
- 在 `handoffs/backend-to-frontend.md` 追加一条记录

### 4. 联调

联调前先回看 `features/` 文档，重点确认：

- 契约是否已经一致
- 是否还有 `open` 阻塞项
- 验收标准是否可以直接验证

### 5. 收口

需求完成后：

- 把 feature 文档状态更新为 `done`
- 把已解决的 handoff 项标记为 `closed`
- 把最终确定的协议或规则补回 feature 文档

## 你们最常用的文件

如果你是前端，最常看：

- `architecture/frontend.md`
- `features/某个需求.md`
- `handoffs/frontend-to-backend.md`

如果你是后端，最常看：

- `architecture/backend.md`
- `features/某个需求.md`
- `handoffs/backend-to-frontend.md`

## 一个实际例子

以 `workflow` 主界面实时消息优化为例：

1. 先看 `features/workflow-realtime-chat.md`
2. 后端先确认：
   - 当前事件流是否支持
   - 新消息事件要长什么样
   - 插入点在哪里
3. 前端先确认：
   - 主消息区怎么展示
   - 是否要 optimistic ack
   - 怎么避免和 task panel 重复
4. 如果前端开发时发现：
   - 后端没有 `workflow_chat_message`
   - 或字段不够
   就写入 `handoffs/frontend-to-backend.md`
5. 如果后端不确定：
   - 主消息区展示多少
   - 是否和任务卡重复
   就写入 `handoffs/backend-to-frontend.md`

## 简化原则

这套文档不要写得太重。

重点只保留三件事：

- 当前需求到底要做什么
- 另一侧现在缺什么
- 最后双方达成了什么结论

如果某条记录已经没人会再看，就不要写成长篇大论。
