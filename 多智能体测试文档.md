# 多智能体测试文档

## 1. 文档目标

本文档用于指导你对当前多智能体实现进行完整测试，覆盖：

- 后端脚本级测试：`backend/test_multi_agent.py`
- 后端 API / 配置测试：`backend/tests/test_custom_agent.py`
- 前端交互联调测试
- 关键场景的预期结果与排障方法

文档默认你当前代码已经是最新实现，并且多智能体主图使用 `multi_agent`。

## 2. 测试范围

建议至少覆盖以下三类：

1. 后端确定性回归
- 验证 Planner / Router / Executor 主链路
- 验证 clarification 中断 / 恢复
- 验证 `planner_goal`、`verified_facts`、`task_pool`

2. 后端管理面回归
- 验证 Agent API 的第一阶段字段
- 验证 `system_prompt_file` 与 prompt 文件写入一致
- 验证配置读取与持久化

3. 前端联调回归
- 验证用户消息提交
- 验证任务卡片的 started / running / completed 展示
- 验证 clarification 中断后的继续交互
- 验证最终答案展示

## 4. 后端脚本测试：test_multi_agent.py

文件：`E:\work\deer-flow\backend\test_multi_agent.py`

### 4.1 测试目的

这个脚本是一个确定性的端到端回归测试，不依赖真实模型与真实领域 Agent，而是用 stub 固定行为，验证：

- 首轮任务拆解
- clarification 中断
- 用户澄清后恢复执行
- 两个子任务全部完成
- `planner_goal` 保持一致
- `verified_facts` 写回 2 条事实
- progress events 正常发出
- 最终 `execution_state = DONE`

### 4.2 执行命令

```powershell
cd E:\work\deer-flow\backend
python test_multi_agent.py
```

如果你希望严格使用虚拟环境：

```powershell
cd E:\work\deer-flow\backend
.\.venv\Scripts\python.exe test_multi_agent.py
```

### 4.3 预期输出

你会看到两轮日志：

1. `=== First turn ===`
- `execution_state: INTERRUPTED`
- `task statuses: ['RUNNING', 'PENDING']`
- `planner_goal` 为首轮问题

2. `=== Resumed turn ===`
- `execution_state: DONE`
- `final_result: Employee ID is A-1001 and there is no active leave record.`
- `verified_facts` 中有 2 条记录
- `task statuses: ['DONE', 'DONE']`
- 末尾输出 `multi_agent integration test passed`

### 4.4 判定标准

通过标准：

- 脚本退出码为 `0`
- 包含 `multi_agent integration test passed`
- 没有 AssertionError

失败时重点看：

- 是否停在 `INTERRUPTED` 后没有进入第二轮
- 是否 `planner_goal` 丢失
- 是否 `verified_facts` 数量不对
- 是否没有 `task_completed` 事件

## 7.3 核心手工测试场景

### 场景 1：普通多任务问答

目的：验证拆解、路由、执行、汇总。

输入示例：

```text
帮我查一下王明天是否请假，以及他的工号
```

预期：

- 页面出现用户消息
- 任务卡片出现，至少 1 个进入 `in_progress`
- 子任务状态逐步变化
- 最终出现汇总答案
- 不应该直接空白结束

重点观察：

- 是否出现多个任务卡片
- 卡片是否会从 `in_progress` 变为 `completed`
- 最终答案是否出现在主消息列表中，而不只是内部状态里

### 场景 2：clarification 中断与恢复

目的：验证 `ask_clarification` 外层中断、用户继续输入后恢复执行。

第一轮输入示例：

```text
帮我查一下王明天是否请假，以及他的工号
```

如果系统进入澄清，继续回复：

```text
研发部的王明天
```

预期：

第一轮：
- 会出现一条澄清消息
- 整个线程不会报错
- 前端不会卡死在 loading

第二轮：
- 原任务继续执行，而不是被当作全新任务
- 任务卡片继续推进
- 最终能得到完整答案

重点观察：

- 澄清消息是否显示在主对话区
- 第二次输入后是否继续原任务，而不是清空成另一个无关回答

### 场景 3：新问题打断旧线程

目的：验证“澄清回复”和“新问题”被正确区分。

步骤：

1. 先发一个会触发多任务的问题
2. 不回复澄清
3. 直接发一个新问题，例如：

```text
今天天气怎么样
```

预期：

- 系统应识别为新轮次
- 旧的 `task_pool` / `verified_facts` 不应继续污染新问题
- 新问题应走新的 Planner 拆解或直接处理

重点观察：

- 不应把“今天天气怎么样”当成上一个问题的澄清回答
- 新问题的最终结果应与旧任务无关

### 场景 4：任务卡片进度可视化

目的：验证 custom events 到前端卡片的链路。

预期：

- 出现 `task_started`
- `task_running` 时卡片状态为加载中
- 卡片折叠态能看到当前进度文案
- 完成后卡片状态变为 completed
- 失败时卡片显示 error

重点观察：

- 运行中时不应始终只显示固定的 `in_progress`
- 现在字符串型进度也应显示在任务卡片中

### 场景 5：Agent 管理面字段验证

如果前端提供 Agent 管理入口，可测试：

- 创建一个 Agent
- 设置：
  - `domain`
  - `system_prompt_file`
  - `hitl_keywords`
  - `max_tool_calls`
- 保存后再读取

预期：

- 字段显示和保存一致
- 后端 `config.yaml` 中有对应字段
- prompt 文件写在 `system_prompt_file` 对应文件中

## 9. 常见问题与排障

### 9.1 pytest 报全局环境错误

现象：

- `python -m pytest` 报 `AttributeError('__spec__')`

原因：

- 你用了全局 Python，不是 `backend/.venv`

解决：

```powershell
cd E:\work\deer-flow\backend
.\.venv\Scripts\python.exe -m pytest tests/test_custom_agent.py -p no:langsmith -p no:cacheprovider
```

### 9.2 PowerShell 里 `pnpm` 被执行策略拦截

现象：

- 报 `PSSecurityException`

解决：

用：

```powershell
pnpm.cmd typecheck
pnpm.cmd lint
```

不要直接写 `pnpm typecheck`。

### 9.3 前端看不到任务卡片更新

重点检查：

- `assistantId` 是否为 `multi_agent`
- 浏览器 Network 是否拿到了 `custom` stream event
- 后端是否有 `task_started/task_running/task_completed`
- 前端是否已是最新代码

### 9.4 看不到最终 AI 答案，只看到内部任务完成

重点检查：

- Planner `DONE` 分支是否把最终总结写进了 `messages`
- 前端消息列表是否渲染最后一条 AIMessage

### 9.5 clarification 后回复没有继续原任务

重点检查：

- 前一条 AIMessage 是否是 `ask_clarification`
- 第二条用户消息是否被错误识别为新问题
- Executor 是否把澄清内容拼进了下一次上下文
