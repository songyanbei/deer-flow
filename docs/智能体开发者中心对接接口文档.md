# 智能体开发者中心对接接口文档

## 1. 文档目的

本文档基于 DeerFlow 当前代码实现整理，目标是给“智能体开发者中心”提供一份可直接对接的接口说明，覆盖：

- 智能体的增删改查
- 智能体在 DeerFlow 中的实际存储方式
- 智能体的实际执行入口
- 多租户/多用户场景下的资源分层
- 对接建议与注意事项

本文档描述的是“当前代码已支持的真实接口”，不是未来规划接口。

---

## 2. 总体对接模型

当前系统中，智能体对接分为两层：

### 2.1 管理面

用于在 DeerFlow 中创建、更新、删除、查询智能体配置。

核心接口：

- `GET /api/agents`
- `GET /api/agents/{name}`
- `POST /api/agents`
- `PUT /api/agents/{name}`
- `DELETE /api/agents/{name}`
- `POST /api/agents/sync`

### 2.2 运行面

用于真正执行智能体。当前 DeerFlow 并没有“直接执行某个 agent 的单独 run 接口”，而是通过 Runtime Thread + SSE 消息流来运行：

- `POST /api/runtime/threads`
- `GET /api/runtime/threads/{thread_id}`
- `POST /api/runtime/threads/{thread_id}/messages:stream`

执行时通过以下字段控制本次可执行的智能体范围：

- `allowed_agents`
- `entry_agent`
- `requested_orchestration_mode`

## 3.1 查询智能体列表

### 接口

`GET /api/agents`

### 说明

- 默认返回当前租户层的智能体列表
- 如果传 `view=merged`，返回三层合并视图：
  - 平台层 `platform`
  - 租户层 `tenant`
  - 个人层 `personal`
- 合并优先级：
  - `personal > tenant > platform`

| 参数 | 必填 | 说明 |
| --- | --- | --- |
| `view` | 否 | 传 `merged` 时返回三层合并结果 |

      "description": "会议助手",
      "model": "gpt-4.1",
      "engine_type": "react",

### 说明

- 列表接口默认不返回 `soul` 内容

## 3.4 创建智能体

### 接口

`POST /api/agents`

### 权限

- 需要 `admin` 或 `owner`

### 请求体

```json
{
  "name": "meeting-agent",
  "description": "会议助手",
  "model": "gpt-4.1",
  "engine_type": "react",
  "tool_groups": ["web", "file:read"],
  "domain": "meeting",
  "system_prompt_file": "SOUL.md",
  "hitl_keywords": ["审批", "确认"],
  "max_tool_calls": 20,
  "mcp_binding": {
    "use_global": false,
    "domain": ["meeting-assistant"],
    "shared": [],
    "ephemeral": []
  },
  "available_skills": ["calendar"],
  "requested_orchestration_mode": "workflow",
  "soul": "你是一个会议助手..."
}
```

### 字段说明

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `name` | 是 | 智能体名称，存储时转小写 |
| `description` | 否 | 智能体描述 |
| `model` | 否 | 模型覆盖配置 |
| `engine_type` | 否 | 执行引擎类型，代码中会做 canonical normalize |
| `tool_groups` | 否 | 工具白名单 |
| `domain` | 否 | 业务域标识，workflow 模式下用于路由发现 |
| `system_prompt_file` | 否 | 提示词文件名，默认 `SOUL.md` |
| `hitl_keywords` | 否 | 人工介入关键词 |
| `max_tool_calls` | 否 | 单次任务最大工具调用次数 |
| `mcp_binding` | 否 | MCP 装配配置 |
| `available_skills` | 否 | 可暴露 skill 白名单 |
| `requested_orchestration_mode` | 否 | 默认编排模式：`auto` / `leader` / `workflow` |
| `soul` | 是 | 系统提示词正文 |

### 成功返回

- HTTP `201`
- 返回创建后的完整智能体信息

### 常见失败

| 状态码 | 说明 |
| --- | --- |
| `409` | 智能体已存在 |
| `422` | 名称非法或 `system_prompt_file` 非法 |
| `500` | 落盘失败 |

## 3.5 更新智能体

### 接口

`PUT /api/agents/{name}`

### 权限

- 需要 `admin` 或 `owner`

### 请求体

字段均可选，按需更新。

```json
{
  "description": "新的描述",
  "model": "gpt-4.1",
  "engine_type": "react",
  "tool_groups": ["web"],
  "domain": "meeting",
  "system_prompt_file": "ACTIVE.md",
  "hitl_keywords": ["审批"],
  "max_tool_calls": 12,
  "mcp_binding": {
    "use_global": false,
    "domain": ["meeting-assistant"],
    "shared": ["time-server"],
    "ephemeral": []
  },
  "available_skills": ["calendar"],
  "requested_orchestration_mode": "leader",
  "soul": "更新后的提示词"
}
```

### 更新行为

- 未传字段保持原值
- 如果修改了 `system_prompt_file`：
  - 系统会更新 `config.yaml`
  - 如果新文件不存在且旧文件存在，会尝试迁移旧 prompt 内容到新文件
- 如果传了 `soul`，系统会把内容写入当前生效的 prompt 文件

### 常见失败

| 状态码 | 说明 |
| --- | --- |
| `404` | 智能体不存在 |
| `422` | 参数非法 |
| `500` | 更新失败 |

## 3.6 删除智能体

### 接口

`DELETE /api/agents/{name}`

### 权限

- 需要 `admin` 或 `owner`

### 说明

- 删除整个智能体目录
- 会连同以下内容一起删除：
  - `config.yaml`
  - prompt 文件
  - `memory.json`
  - 该 agent 目录下其他文件

### 返回

- HTTP `204`

## 3.7 批量同步智能体

### 接口

`POST /api/agents/sync`

### 权限

- 需要 `admin` 或 `owner`

### 使用场景

如果开发者中心是“智能体主数据源”，推荐优先使用这个接口，而不是前端逐条调用创建/更新。

### 请求体

```json
{
  "mode": "upsert",
  "agents": [
    {
      "name": "meeting-agent",
      "description": "会议助手",
      "model": "gpt-4.1",
      "engine_type": "react",
      "tool_groups": ["web"],
      "domain": "meeting",
      "system_prompt_file": "SOUL.md",
      "hitl_keywords": ["审批"],
      "max_tool_calls": 20,
      "mcp_binding": {
        "use_global": false,
        "domain": ["meeting-assistant"],
        "shared": [],
        "ephemeral": []
      },
      "available_skills": ["calendar"],
      "requested_orchestration_mode": "workflow",
      "soul": "你是一个会议助手"
    }
  ]
}
```

### `mode` 说明

| 值 | 含义 |
| --- | --- |
| `upsert` | 只对请求中的 agent 做创建或更新，不删除其他 agent |
| `replace` | 先创建/更新请求中的 agent；如果全部成功，再删除未出现在请求中的其他 agent |

### 返回示例

```json
{
  "created": ["meeting-agent"],
  "updated": [],
  "deleted": [],
  "errors": []
}
```

### 关键注意事项

- `replace` 模式下，如果本次同步中有任一 agent upsert 失败，系统不会执行删除，避免误删存量数据
- 请求内不允许出现重复名称，否则直接 `422`

## 5. 智能体实际存储方式

## 5.1 单个智能体目录结构

每个智能体最终落盘为一个目录：

```text
agents/{agent_name}/
  config.yaml
  SOUL.md
  memory.json
```

其中：

- `config.yaml`：智能体结构化配置
- `SOUL.md` 或指定 prompt 文件：系统提示词正文
- `memory.json`：该智能体自己的记忆数据

---

## 5.2 多层存储结构

当前代码支持三层智能体来源：

### 平台层

```text
{base_dir}/agents/{agent_name}/
```

### 租户层

```text
{base_dir}/tenants/{tenant_id}/agents/{agent_name}/
```

### 用户个人层

```text
{base_dir}/tenants/{tenant_id}/users/{user_id}/agents/{agent_name}/
```

### 合并优先级

同名智能体按以下优先级覆盖：

`personal > tenant > platform`

## 6. 智能体执行接口

## 6.1 执行模型说明

当前 DeerFlow 的执行入口不是“直接运行某个智能体”，而是：

1. 先创建 runtime thread
2. 再向 thread 提交消息
3. 在请求中指定本次允许参与执行的智能体集合
4. DeerFlow 根据编排模式决定走 `leader` 或 `workflow`
5. 真正执行时再按 agent 配置装载 prompt、model、tools、MCP、memory

所以开发者中心如果要“运行某个智能体”，推荐做法是：

- `allowed_agents` 只传该智能体
- `entry_agent` 传该智能体名

这样 DeerFlow 的实际执行范围就会被限制到这个 agent。

## 6.2 创建运行线程

### 接口

`POST /api/runtime/threads`

### 请求体

```json
{
  "portal_session_id": "dev-center-session-001"
}
```

### 返回示例

```json
{
  "thread_id": "4d4ab1df-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
  "portal_session_id": "dev-center-session-001",
  "tenant_id": "tenant-a",
  "user_id": "u-001",
  "created_at": "2026-04-13T09:00:00+00:00"
}
```

### 说明

- `portal_session_id` 是开发者中心侧会话 ID
- DeerFlow 返回 `thread_id` 作为后续执行的主键
- 系统会在本地 registry 中记录 thread 与 tenant/user/binding 的关联

## 6.4 提交消息并流式执行

### 接口

`POST /api/runtime/threads/{thread_id}/messages:stream`

### Content-Type

`application/json`

### 返回类型

`text/event-stream`

### 请求体

```json
{
  "message": "帮我整理下今天的会议纪要",
  "group_key": "meeting-group",
  "allowed_agents": ["meeting-agent"],
  "entry_agent": "meeting-agent",
  "requested_orchestration_mode": "workflow",
  "metadata": {
    "scene": "developer-center",
    "agent_version": "v1"
  }
}
```

### 字段说明

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `message` | 是 | 用户消息 |
| `group_key` | 是 | 业务分组键 |
| `allowed_agents` | 是 | 本次允许运行的 agent 名单，不能为空 |
| `entry_agent` | 否 | 指定入口 agent，必须包含在 `allowed_agents` 中 |
| `requested_orchestration_mode` | 否 | `auto` / `leader` / `workflow` |
| `metadata` | 否 | 只允许 primitive 值：string/number/bool/null |

### 校验规则

- `allowed_agents` 中每个 agent 都必须能通过当前三层配置解析成功
- `entry_agent` 必须在 `allowed_agents` 里
- `message` / `group_key` 不能为空
- 单次 `allowed_agents` 最多 100 个

## 6.5 SSE 事件

当前对外暴露的稳定事件名包括：

- `ack`
- `message_delta`
- `message_completed`
- `artifact_created`
- `intervention_requested`
- `governance_created`
- `run_completed`
- `run_failed`

### `ack`

```text
event: ack
data: {"thread_id":"..."}
```

### `message_delta`

表示模型流式文本片段。

### `message_completed`

表示一条完整回复结束。

### `artifact_created`

表示执行中产生了工件，例如文件、输出结果等。

### `intervention_requested`

表示命中了人工介入、审批、澄清等中断点。

### `governance_created`

表示生成了治理/审批相关记录。

### `run_completed`

表示本次执行完成。

### `run_failed`

表示本次执行失败。

## 7. DeerFlow 中一次执行是如何落到实际智能体上的

## 7.1 执行前

Runtime 接口收到消息后，会把以下信息作为上下文传给 LangGraph：

- `thread_id`
- `tenant_id`
- `user_id`
- `allowed_agents`
- `group_key`
- `requested_orchestration_mode`
- `agent_name`（如果传了 `entry_agent`）
- `thread_context`

## 7.2 Agent 构建时

真正构建 agent 时，系统会按 `agent_name + tenant_id + user_id` 去三层配置中解析 agent：

- 读取 `config.yaml`
- 读取 `SOUL.md` 或指定 prompt 文件
- 读取 per-agent memory
- 解析 model 覆盖配置
- 解析 `tool_groups`
- 解析 `available_skills`
- 解析 `mcp_binding`
- 注入 runbook / persistent memory / MCP tools

## 7.3 Workflow 模式下

如果走 workflow，executor 会：

- 根据 planner/router 分配 `assigned_agent`
- 按 agent 名预热 MCP
- 为该任务构造 domain-agent 运行配置
- 用 `make_lead_agent(agent_config_override)` 实例化实际执行 agent
- 让该 domain agent 完成任务

也就是说，开发者中心维护的 agent 配置，最终会在执行瞬间被真实装配进模型、工具、MCP 和 prompt。

## 8. 推荐对接方案

## 8.1 如果开发者中心是“管理主系统”

推荐：

1. 智能体配置管理使用 `POST /api/agents/sync`
2. 单条调试时使用单个 CRUD 接口
3. 运行时统一走 runtime thread + stream 接口

这样可以把“配置同步”和“执行调用”解耦。

## 8.2 如果开发者中心要“执行某个指定智能体”

推荐请求方式：

```json
{
  "message": "xxx",
  "group_key": "agent-debug",
  "allowed_agents": ["meeting-agent"],
  "entry_agent": "meeting-agent",
  "requested_orchestration_mode": "leader"
}
```

如果希望 DeerFlow 在多个领域 agent 之间做编排，则：

- `allowed_agents` 传多个 agent
- `entry_agent` 可不传或传主入口 agent
- `requested_orchestration_mode` 传 `workflow`

## 8.3 如果需要支持用户个人智能体

当前代码还提供了个人资源接口：

- `GET /api/me/agents`
- `POST /api/me/agents`
- `GET /api/me/agents/{name}`
- `PUT /api/me/agents/{name}`
- `DELETE /api/me/agents/{name}`

这套接口适合“登录用户自己管理自己的 agent”，不适合作为租户统一开发者中心的主接口。

---

## 9. 关键注意事项

## 9.1 当前执行接口不是 agent 直调

外部系统不要假设存在：

- `POST /api/agents/{name}/run`

当前真实入口是 Runtime Thread + SSE Stream。

## 9.2 `allowed_agents` 是强约束

执行时只有出现在 `allowed_agents` 里的智能体才允许参与本次运行。

## 9.3 `entry_agent` 不是必填，但很重要

如果开发者中心想明确“这次就是从某个 agent 开始执行”，建议显式传 `entry_agent`。

## 9.4 名称会统一转小写

开发者中心侧如果保存 agent code / key，建议也统一使用小写。

## 9.5 删除是目录级删除

删除智能体会直接删除整个 agent 目录及其内部文件，包含 memory。

## 9.6 `view=merged` 适合展示，不适合直接作为编辑目标

因为 merged 结果可能来自 platform / tenant / personal 任一层。开发者中心如需编辑，最好明确编辑的是哪一层。

---

## 10. 建议给开发者中心落的最小对接闭环

建议最小闭环如下：

1. 列表页：`GET /api/agents`
2. 创建页：`POST /api/agents`
3. 编辑页：`GET /api/agents/{name}` + `PUT /api/agents/{name}`
4. 删除：`DELETE /api/agents/{name}`
5. 调试运行：
   - `POST /api/runtime/threads`
   - `POST /api/runtime/threads/{thread_id}/messages:stream`
6. 批量发布：
   - `POST /api/agents/sync`

### 12.1 单条更新 MCP 服务器

```
PUT /api/mcp/config/{name}
```

请求体：`McpServerConfigResponse` JSON（同 GET 返回的 schema），新增字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `source` | `string \| null` | 来源标记（如 `moss-portal`），用于所有权门控 |
| `mcp_kind` | `string \| null` | `local` / `remote`，透传字段供平台过滤 |

**所有权规则**：若已存在同名条目且 `source` 不匹配 → 409 Conflict。`source=null` 的条目可被任何来源覆盖。

响应 200：

```json
{ “name”: “github”, “source”: “moss-portal”, “updated_at”: “2026-04-16T...” }
```

### 12.2 单条删除 MCP 服务器

```
DELETE /api/mcp/config/{name}?source=moss-portal
```

- `source` 参数必须与现存条目的 `source` 一致，否则 409
- 不传 `source` 且现存条目有 `source` → 403
- 条目不存在 → 404

响应 200：`{ “deleted”: “github” }`

### 12.3 整表 PUT（保留，行为不变）

```
PUT /api/mcp/config
```

内部已走 per-tenant 互斥锁 + 原子写，行为与此前一致，不强制 `source`。

## 13. Skill 直传安装接口（资源同步新增）

### 13.1 上传安装 — install_from_payload

```
POST /api/skills/install_from_payload
Content-Type: multipart/form-data
```

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `file` | binary | 是 | .skill ZIP 归档文件 |
| `source` | string | 否 | 来源标记（写入 `.install_meta.json`） |
| `overwrite` | bool | 否 | 是否覆盖已有同名 skill，默认 false |

大小上限：50MB（可通过环境变量 `SKILL_PAYLOAD_MAX_BYTES` 调整）。

响应 200：

```json
{ “success”: true, “skill_name”: “calendar”, “message”: “...”, “source”: “moss-portal” }
```

### 13.2 URL 下载安装 — install_from_url

```
POST /api/skills/install_from_url
Content-Type: application/json
```

```json
{
  “url”: “https://platform.example.com/skills/calendar-1.0.0.skill”,
  “source”: “moss-portal”,
  “checksum_sha256”: “abc...”,
  “overwrite”: false
}
```

SSRF 防护：
- 仅允许 `https://`
- host 必须在环境变量 `SKILL_SOURCE_ALLOWLIST`（逗号分隔）白名单中
- 不允许跨 host 重定向
- 下载大小上限 50MB，超时 30s
- SHA-256 校验必须通过，否则 422

## 14. Agent Sync 依赖校验（资源同步新增）

`POST /api/agents/sync` 请求体新增字段：

| 字段 | 类型 | 默认 | 说明 |
|------|------|------|------|
| `validate_dependencies` | bool | true | 同步前校验 mcp_binding 和 available_skills 引用是否存在 |

**校验行为**：

- `mcp_binding.domain / shared / ephemeral` 中每个 server name 必须在合并后的 `extensions_config.mcp_servers` 中存在
- `available_skills` 中每个 skill name 必须在租户合并后的 skills 集合中存在
- 校验失败 → 该 agent 的 `AgentSyncItemResult.action=”failed”`，附带具体缺失资源名
- 单 agent 失败不阻塞批次中其他 agent
- 设置 `validate_dependencies: false` 可跳过（适用于”先推 agent 后推资源”的乱序场景）

## 15. 并发安全与原子写

所有 MCP / Skill / Agent 配置写入操作均已走：

1. **per-tenant 异步互斥锁**：同一租户的同一资源类型（mcp/skill/agent）写入串行化
2. **文件级 advisory lock**：跨 worker（多 gunicorn 进程）互斥
3. **原子写**：写临时文件 → fsync → `os.replace`，崩溃不会留下半截文件

## 16. 一句话结论

如果”智能体开发者中心”负责智能体配置管理，那么 DeerFlow 当前已经具备完整承接能力：

- 管理侧用 `/api/agents` 和 `/api/agents/sync`（支持依赖校验）
- MCP 资源同步用 `PUT/DELETE /api/mcp/config/{name}`（支持 source 所有权门控）
- Skill 直传用 `/api/skills/install_from_payload` 和 `/api/skills/install_from_url`
- 执行侧用 `/api/runtime/threads` 和 `/messages:stream`
- 存储侧由 DeerFlow 自动落盘到 platform/tenant/personal 三层目录，并在运行时按优先级解析和装配
- 所有写入操作均有 per-tenant 并发锁 + 原子写保护
