# DeerFlow 部署与使用指南

本文档面向新加入项目的开发者，目标是让同事拿到仓库后，能够基于本文档独立完成环境准备、配置、启动、验证、日常使用和故障排查。

## 1. 项目概览

DeerFlow 是一个基于 LangGraph 和 LangChain 的多能力 AI Agent Harness，当前仓库是一个完整的前后端项目，核心由以下部分组成：

- `frontend/`：Next.js 16 + React 19 前端工作台，默认开发端口 `3000`
- `backend/`：Python 3.12 + FastAPI + LangGraph 后端，包含 Agent、工具、记忆、文件、上传、MCP、子代理等能力
- `docker/`：Docker 开发编排与 Nginx 反向代理配置
- `skills/`：公共技能与自定义技能目录
- `config.example.yaml` / `config.yaml`：主配置文件，定义模型、工具、沙箱、记忆等
- `.env`、`frontend/.env`：后端和前端环境变量

统一访问入口如下：

- 前端入口：`http://localhost:2026`
- Gateway API：`http://localhost:2026/api/*`
- LangGraph API：`http://localhost:2026/api/langgraph/*`
- Swagger：`http://localhost:2026/docs`
- 健康检查：`http://localhost:2026/health`

## 2. 运行架构

项目在开发态下通常包含 4 个服务：

1. `frontend`
   负责页面渲染、聊天交互、设置页面、任务面板和产物展示。
2. `gateway`
   提供模型列表、技能管理、MCP 配置、记忆、上传和产物下载等 REST API。
3. `langgraph`
   提供 Agent 执行、线程、流式输出、多代理编排等核心能力。
4. `nginx`
   统一反向代理，将 `/` 转发到前端、`/api/*` 转发到 Gateway、`/api/langgraph/*` 转发到 LangGraph。

当 `config.yaml` 使用 `src.community.aio_sandbox:AioSandboxProvider` 且配置了 `provisioner_url` 时，还会额外启动：

5. `provisioner`
   用于 Kubernetes 模式下的沙箱生命周期管理。

## 3. 部署方式选择

### 3.1 推荐方式：Docker 开发部署

适用场景：

- 新同事首次启动项目
- 希望最少依赖、最快跑通项目
- Linux、macOS、Windows + WSL / Git Bash / Docker Desktop

优点：

- 启动路径统一
- 自带 Nginx、前端、Gateway、LangGraph
- 更接近团队一致环境

注意：

- 当前仓库提供的是“开发/联调部署编排”，不是严格意义上的生产级编排
- `docker/docker-compose-dev.yaml` 中前端和后端都以开发模式启动，适合开发、联调、测试，不建议直接视为正式生产方案

### 3.2 次优方式：本地开发部署

适用场景：

- 需要本机热更新
- 需要调试 Python / Next.js 源码
- Linux / macOS / WSL 环境齐全

额外要求：

- 本机需安装 `nginx`
- `make dev` 使用了 `bash`、`pkill`、`lsof` 等命令，原生 Windows PowerShell 不适合作为首选运行环境

### 3.3 Windows 兼容方式：PowerShell 直接启动

适用场景：

- Windows 原生开发
- 临时联调，不依赖 Nginx

特点：

- 通过 `scripts/start-full-dev.ps1` 或 `start-dev.bat` 直接拉起前后端
- 默认不走 Nginx 聚合入口
- 访问地址改为前端 `http://127.0.0.1:3000`，前端会直接请求 `8001` 和 `2024`

## 4. 环境准备

### 4.1 基础依赖

推荐版本：

- Git
- Docker Desktop 4.x+ 与 Docker Compose v2
- Node.js `22+`
- pnpm `10.26.2+`
- Python `3.12+`
- uv
- Nginx

其中：

- 如果使用 Docker 部署，最少需要 Git、Docker Desktop、基础 API Key
- 如果使用本地部署，需要 Node.js、pnpm、Python、uv、Nginx 全部安装到位

### 4.2 克隆仓库

```bash
git clone <your-repo-url>
cd deer-flow
```

### 4.3 初始化配置文件

Linux / macOS / WSL / Git Bash 推荐：

```bash
make config
```

如果 `config.yaml` 已存在，上述命令会中止，这是正常保护行为。你也可以手工确认以下文件存在：

- 根目录 `.env`
- 根目录 `config.yaml`
- `frontend/.env`

## 5. 核心配置说明

### 5.1 最低可运行配置

项目至少需要在 `config.yaml` 中启用一个模型。建议从一个最简单的 OpenAI 模型开始：

```yaml
models:
  - name: gpt-4o
    display_name: GPT-4o
    use: langchain_openai:ChatOpenAI
    model: gpt-4o
    api_key: $OPENAI_API_KEY
    max_tokens: 4096
    temperature: 0.7
    supports_vision: true

tool_groups:
  - name: web
  - name: file:read
  - name: file:write
  - name: bash

sandbox:
  use: src.sandbox.local:LocalSandboxProvider
```

说明：

- `models`：定义可在前端选择的模型列表
- `api_key: $OPENAI_API_KEY`：表示从环境变量中读取，不要把真实密钥直接写入仓库
- `sandbox.use`：默认本地沙箱，最容易跑通

### 5.2 `.env` 建议配置

根目录 `.env` 最少建议配置：

```bash
OPENAI_API_KEY=your-openai-api-key
TAVILY_API_KEY=your-tavily-api-key
JINA_API_KEY=your-jina-api-key
```

可选配置：

- `FIRECRAWL_API_KEY`
- `VOLCENGINE_API_KEY`
- `DEEPSEEK_API_KEY`
- `NOVITA_API_KEY`
- `GEMINI_API_KEY`
- `CONTACTS_MCP_ENTRY`
- `MEETING_ASSISTANT_MCP_ENTRY`
- `TIME_SERVER_MCP_ENTRY`

说明：

- 没有配置某个供应商密钥，不影响项目启动，但相关模型或工具无法工作
- `TAVILY_API_KEY`、`JINA_API_KEY` 缺失时，部分搜索/网页抓取能力会受限
- 如果启用了 workflow/domain agent，并且它依赖外部 MCP 脚本，需要在 `.env` 中补齐对应入口脚本路径

示例：

```bash
CONTACTS_MCP_ENTRY=E:/work/laifu-meeting-agent/mcp-servers/contacts/src/index.js
MEETING_ASSISTANT_MCP_ENTRY=E:/work/laifu-meeting-agent/mcp-servers/meeting-assistant/src/index.js
TIME_SERVER_MCP_ENTRY=E:/work/laifu-meeting-agent/mcp-servers/time-server/src/index.js
```

### 5.3 `frontend/.env`

大多数情况下保持默认即可，不需要主动修改。

只有在“不走 Nginx、直接连接后端”时，才需要启用：

```bash
NEXT_PUBLIC_BACKEND_BASE_URL="http://127.0.0.1:8001"
NEXT_PUBLIC_LANGGRAPH_BASE_URL="http://127.0.0.1:2024"
```

## 6. 推荐部署流程：Docker

### 6.1 首次启动

在仓库根目录执行：

```bash
make config
make docker-init
make docker-start
```

说明：

- `make docker-init` 会预拉取沙箱镜像，减少首次任务执行等待时间
- `make docker-start` 会根据 `config.yaml` 自动判断是否需要 `provisioner`

启动完成后访问：

```text
http://localhost:2026
```

### 6.2 查看日志

```bash
make docker-logs
make docker-logs-frontend
make docker-logs-gateway
```

如果需要看 `provisioner`，可直接执行：

```bash
cd docker
docker compose -p deer-flow-dev -f docker-compose-dev.yaml logs -f provisioner
```

### 6.3 停止服务

```bash
make docker-stop
```

## 7. 本地开发部署流程

### 7.1 检查依赖

```bash
make check
```

此命令会检查：

- Node.js
- pnpm
- uv
- nginx

### 7.2 安装依赖

```bash
make install
```

它会执行：

- `backend/` 下的 `uv sync`
- `frontend/` 下的 `pnpm install`

### 7.3 启动服务

```bash
make dev
```

成功后统一入口为：

```text
http://localhost:2026
```

本地日志位于：

- `logs/langgraph.log`
- `logs/gateway.log`
- `logs/frontend.log`
- `logs/nginx.log`

### 7.4 停止服务

```bash
make stop
```

清理日志与残留：

```bash
make clean
```

## 8. Windows 原生启动流程

如果你不在 WSL / Git Bash 中，而是在 Windows 原生 PowerShell 环境中开发，建议使用项目自带脚本。

### 8.1 推荐脚本

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\start-full-dev.ps1
```

停止服务：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\stop-full-dev.ps1
```

默认地址：

- Frontend：`http://127.0.0.1:3000`
- Gateway：`http://127.0.0.1:8001`
- LangGraph：`http://127.0.0.1:2024`

说明：

- 该方式默认 `mode = direct-no-nginx`
- 运行状态会写入 `.dev-runtime/services.json`
- 这类运行态文件通常不应提交到 Git

### 8.2 备用批处理

```bat
start-dev.bat
```

注意：

- 该脚本会分别打开多个窗口
- 文案里展示了 `http://localhost:2026`，但脚本本身并不会启动 Nginx，所以实际更稳妥的访问方式仍是 `3000 / 8001 / 2024`

## 9. 部署后验证清单

建议按以下顺序验证：

1. 打开 `http://localhost:2026` 或 `http://127.0.0.1:3000`
2. 打开 `http://localhost:2026/health`，确认 Gateway 健康检查正常
3. 打开 `http://localhost:2026/docs`，确认 FastAPI 文档页正常
4. 访问 `http://localhost:2026/api/models`，确认能返回模型列表
5. 在前端新建对话，选择一个已配置模型，发送一条简单消息
6. 上传一个文件，确认上传列表和对话上下文可见
7. 在对话中触发生成产物，确认产物可在前端或线程目录中看到

## 10. 日常使用说明

### 10.1 开发者首次使用

启动完成后：

1. 打开工作台
2. 检查右上或模型选择区域能否看到你在 `config.yaml` 中配置的模型
3. 发起一条简单对话，例如“帮我总结这个项目”
4. 如需文件处理，上传文件后再发起任务

### 10.2 线程与文件目录

运行过程中，线程相关数据会写到：

- `backend/.deer-flow/threads/`

每个线程通常包含：

- `uploads/`：上传文件
- `workspace/`：中间工作目录
- `outputs/`：最终产物

如果使用了沙箱模式，容器中的虚拟路径会映射到这些目录。

### 10.3 技能目录

技能默认从以下目录加载：

- `skills/public/`
- `skills/custom/`

新增自定义技能时，建议按 `某技能目录/SKILL.md` 的结构组织。

### 10.4 Workflow Agent 配置目录

当前仓库已将 workflow/domain agent 的共享定义放在：

- `backend/.deer-flow/agents/<agent-name>/config.yaml`
- `backend/.deer-flow/agents/<agent-name>/SOUL.md`

这些文件适合版本化，便于团队共享。

但以下内容仍然属于运行态数据，不建议提交：

- `backend/.deer-flow/threads/`
- `backend/.deer-flow/memory.json`
- `backend/.deer-flow/agents/*/memory.json`

如果某个 agent 依赖仓库外的 MCP 脚本，请通过 `.env` 中的入口变量注入真实路径，而不要在 `config.yaml` 里写死个人机器绝对路径。

## 11. 常见问题排查

### 11.1 `make` 在 Windows PowerShell 下不可用

原因：

- 根目录 `Makefile` 和 `scripts/docker.sh` 假设存在 Bash 环境

解决：

- 优先使用 WSL / Git Bash
- 或改用 `scripts/start-full-dev.ps1`

### 11.2 打开 `localhost:2026` 失败

排查顺序：

1. 确认 Nginx 是否已启动
2. 确认 `frontend`、`gateway`、`langgraph` 三个服务端口是否都已监听
3. 查看对应日志文件

如果是 Windows 原生脚本启动，优先访问：

- `http://127.0.0.1:3000`

### 11.3 页面能打开，但没有模型可选

原因通常是：

- `config.yaml` 没有启用任何有效模型
- 对应 API Key 未设置
- `config.yaml` 格式错误，导致后端未正确加载配置

建议：

- 先保留一个最简单的 OpenAI 模型配置跑通，再逐步增加其他模型

### 11.4 搜索/抓取相关功能不可用

通常是以下原因之一：

- `TAVILY_API_KEY` 未配置
- `JINA_API_KEY` 未配置
- 网络环境无法访问相应服务

### 11.5 Docker 模式下 `provisioner` 启动异常

仅在 Kubernetes 沙箱模式下需要关注。

检查项：

- `config.yaml` 是否将 `sandbox.use` 设置为 `src.community.aio_sandbox:AioSandboxProvider`
- 是否设置了 `provisioner_url`
- 主机上是否存在可用的 kubeconfig
- `DEER_FLOW_ROOT` 是否正确

### 11.6 仓库里出现 `.dev-runtime/services.json` 变更

这是运行态文件，PowerShell 启动脚本会刷新：

- 启动时间
- PID
- 日志路径

通常不建议作为功能代码提交。

## 12. 团队协作建议

建议团队统一遵循以下实践：

1. 新同事默认使用 Docker 方案启动
2. 本地深度调试时再切换到 `make dev`
3. API Key 统一通过 `.env` 或系统环境变量注入
4. 不要提交运行态文件、临时压缩包和本地产物
5. 新增能力时优先补充到 `config.example.yaml`、`.env.example` 和本文档

## 13. 当前仓库适合的交付方式

基于当前仓库现状，最适合团队内部的交付方式是：

- 开发联调：优先 `make docker-start`
- 源码调试：`make dev`
- Windows 原生：`scripts/start-full-dev.ps1`

如果后续需要“真正的生产部署文档”，建议在当前基础上再补一版生产方案，至少包含以下内容：

- 前端改为 `next build && next start`
- 后端关闭开发热重载
- 独立的生产 `docker-compose.yml` 或 K8s 清单
- 域名、HTTPS、持久化卷、日志采集、监控告警、密钥注入策略

在当前仓库能力范围内，本文档已经覆盖了其他开发者顺利完成部署和日常使用所需的核心步骤。
