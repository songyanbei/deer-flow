# 当前多智能体实现说明

## 1. 目标与范围

本文档描述当前代码库中已落地的多智能体第一阶段实现，覆盖以下内容：

- 多智能体执行图的整体结构
- Planner / Router / Executor 的职责分工
- 共享黑板状态设计
- Agent 注册中心与管理 API
- 中断恢复与多轮对话处理
- 前端任务进度可视化链路
- 已有测试与验证方式

当前实现目标是满足《多智能体架构改造实现方案.md》和《多智能体架构技术路线说明书.md》中第一阶段的核心需求。

## 2. 整体架构

当前后端使用 LangGraph 构建 `multi_agent` 执行图，核心节点如下：

1. `planner`
- 负责任务拆解
- 负责在任务完成后做目标校验
- 负责输出最终结果

2. `router`
- 从 `task_pool` 中选择下一个待执行任务
- 为任务分配对应领域 Agent
- 维护 `route_count`，避免异常循环

3. `executor`
- 调用被分配的领域 Agent 执行子任务
- 写回 `verified_facts`
- 处理 `ask_clarification` 中断与恢复
- 发出前端可消费的任务进度事件

## 3. 共享黑板状态

共享状态定义在 `backend/src/agents/thread_state.py`，关键字段如下：

- `original_input`
  - 当前轮用户原始问题
- `planner_goal`
  - Planner 归属的本轮总目标
  - 当前实现中与首轮用户问题等价，但已独立进入黑板
- `task_pool`
  - 当前轮待执行/执行中/已完成的任务列表
- `verified_facts`
  - 结构化事实表，按 `task_id` 写回子任务结果
- `route_count`
  - 路由计数，防止异常循环
- `execution_state`
  - 执行阶段状态，例如 `PLANNING_DONE`、`INTERRUPTED`、`DONE`
- `final_result`
  - 最终汇总结果

### 3.1 task_pool reducer

`task_pool` 使用带状态迁移保护的 reducer：

- 允许的状态迁移：
  - `PENDING -> RUNNING/FAILED`
  - `RUNNING -> DONE/FAILED`
- 拒绝非法回退：
  - 例如 `RUNNING -> PENDING`
  - 例如 `DONE -> PENDING`
- 支持 `[]` 显式清空任务池

### 3.2 verified_facts reducer

`verified_facts` 不是简单字符串列表，而是可合并的键值事实表：

```python
{
  "task_id": {
    "agent": "contacts-agent",
    "task": "lookup employee id",
    "summary": "Employee ID is A-1001"
  }
}
```

这样可以支持：

- 任务级结果索引
- 后续校验时按任务读取事实
- 后续阶段继续扩展来源、时间戳、结构化字段

## 4. Planner 实现

文件：`backend/src/agents/planner/node.py`

### 4.1 首轮任务拆解

首次进入 Planner 时：

- 读取 `original_input`
- 调用 `DECOMPOSE_SYSTEM_PROMPT`
- 生成一组 `PENDING` 子任务写入 `task_pool`
- 写入 `planner_goal`
- 重置 `route_count = 0`

### 4.2 任务完成后的目标校验

当本轮任务都结束后，Planner 会进入 validate 模式：

- 汇总 `task_pool`
- 汇总 `verified_facts`
- 使用 `planner_goal` 作为校验目标
- 调用 `VALIDATE_SYSTEM_PROMPT`
- 如果目标完成：
  - 写 `execution_state = DONE`
  - 写 `final_result`
  - 同步向 `messages` 追加最终 AI 回复

### 4.3 多轮对话与新轮次识别

Planner 现在支持区分：

1. 澄清回复
- 如果上一条 AI 消息是 `ask_clarification`
- 当前用户消息被视为同一轮恢复输入
- 不重置黑板

2. 新问题
- 如果最新 HumanMessage 与 `original_input` 不同
- 且前一条不是澄清消息
- 视为新一轮对话
- 重置：
  - `task_pool`
  - `verified_facts`
  - `original_input`
  - `planner_goal`
  - `route_count`
  - `final_result`

## 5. Router 实现

Router 基于当前 `task_pool` 中的 `PENDING` 任务进行分配，领域 Agent 来源于注册中心中的 `domain` 配置。

当前 Router 能力包括：

- 选择下一条待执行任务
- 为任务绑定 `assigned_agent`
- 将任务状态推进到 `RUNNING`
- 维护 `route_count`
- 在无法继续分配时走系统兜底逻辑

## 6. Executor 实现

文件：`backend/src/agents/executor/executor.py`

### 6.1 子任务执行

Executor 的执行流程：

1. 从 `task_pool` 中取第一个 `RUNNING` 任务
2. 按 `assigned_agent` 构造领域 Agent 运行配置
3. 构建任务上下文
   - 子任务描述
   - `verified_facts`
   - 用户澄清补充信息
4. 调用领域 Agent
5. 根据结果更新：
   - `task_pool`
   - `verified_facts`
   - `execution_state`

### 6.2 clarification 中断恢复

当领域 Agent 返回 `ask_clarification` 时：

- Executor 不再把子图 `ToolMessage` 直接抛给外层
- 而是写入外层 `AIMessage(name="ask_clarification")`
- 设置 `execution_state = INTERRUPTED`
- 图在外层中断，等待用户下一条消息

恢复时：

- Executor 会检查最近两条消息
- 若前一条为 `ask_clarification`，后一条 HumanMessage 会被视为澄清回答
- 澄清内容会拼入下一次领域 Agent 的上下文

### 6.3 verified_facts 回写

任务完成后，Executor 会按 `task_id` 将结构化摘要写入 `verified_facts`，供后续任务和 Planner 校验使用。

## 7. Agent 注册中心与管理 API

文件：`backend/src/gateway/routers/agents.py`

当前管理 API 已完整支持第一阶段编排字段：

- `domain`
- `system_prompt_file`
- `hitl_keywords`
- `max_tool_calls`

### 7.1 create / update 行为

创建或更新 Agent 时：

- 会把上述字段写入 `config.yaml`
- Prompt 内容不再固定写入 `SOUL.md`
- 而是写入 `system_prompt_file` 指定文件

### 7.2 prompt 文件迁移

当已有 Agent 更新 `system_prompt_file` 时：

- 如果只换文件名、不传新 `soul`
- 系统会把旧 prompt 内容迁移到新文件
- 避免出现“管理面已切换配置，但运行时 prompt 丢失”的情况

## 8. 领域 Agent 运行时约束

### 8.1 Domain Agent 中间件裁剪

领域 Agent 调用时设置 `is_domain_agent = True`，因此会跳过：

- `TitleMiddleware`
- `MemoryMiddleware`

这样可以避免子任务执行过程中对线程标题和长期记忆产生不必要副作用。

### 8.2 Tool 调用上限

新增 `ToolCallLimitMiddleware`，结合 `AgentConfig.max_tool_calls` 生效。

作用：

- 防止单个领域 Agent 在一个子任务中无限调用工具
- 将设计文档中的上限约束从配置项落到运行时行为

## 9. 前端任务进度链路

涉及文件：

- `frontend/src/core/threads/hooks.ts`
- `frontend/src/core/tasks/context.tsx`
- `frontend/src/components/workspace/messages/subtask-card.tsx`

### 9.1 后端事件

Executor 当前发出的自定义事件包括：

- `task_started`
- `task_running`
- `task_completed`
- `task_failed`

### 9.2 前端事件消费

前端 `useThreadStream` 会消费这些 custom events，并更新 `SubtaskContext`。

### 9.3 卡片展示

`SubtaskCard` 当前支持展示：

- 任务标题
- 当前状态
- 运行中字符串进度提示
- 子任务完成结果
- 子任务失败信息

这使得多智能体链路不再只有最终答复可见，而是具备任务级执行过程可视化能力。

## 11. 当前验收结论

基于当前代码与回归结果，第一阶段已满足以下关键目标：

- 多智能体执行图已替换单 Lead Agent 主链路
- Planner / Router / Executor 职责明确
- 黑板状态具备任务池、事实表、目标字段与状态保护
- 多轮对话与澄清中断恢复正确
- Agent 注册中心管理面支持第一阶段新增字段
- Prompt 文件配置与运行时行为一致
- 前端可见任务进度和最终结果
- 后端集成测试与 API 测试均通过

## 12. 当前仍可继续优化的方向

以下属于后续增强项，不影响当前第一阶段验收：

1. 更强的 `planner_goal` 归一化
- 当前等价于首轮用户输入
- 后续可以改为 Planner 生成的标准化目标文本

2. 更丰富的 `verified_facts` 结构
- 可扩展来源、时间戳、置信度等字段

3. 更细粒度的前端进度事件
- 例如区分 dispatch、tool-use、waiting-approval 等阶段

4. 更多端到端回归场景
- 新问题打断旧线程
- `SYSTEM_FALLBACK`
- 工具上限触发
- MCP 连接失败恢复
