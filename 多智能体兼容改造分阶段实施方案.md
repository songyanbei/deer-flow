# 多智能体兼容改造分阶段实施方案

> 基于当前 `deer-flow` 代码现状，面向“原生 `leader agent` 模式与新 `multi_agent workflow` 模式共存、前端统一展示、运行时可自动分流”的落地实施文档。
>
> 文档版本：v1.1  
> 编写日期：2026-03-09  
> 当前状态：阶段 0-2 已完成，阶段 3 起进入模式共存与产品化落地阶段

---

## 1. 背景与目标

当前代码库中已经同时存在两套任务协作机制：

1. 原生 `leader agent` / 旧 `subagent` 机制
- 由 `lead_agent` 在对话过程中调用 `task` 工具触发
- 前端通过消息分组 `assistant:subagent` 渲染任务卡片
- 本质是“主代理驱动、消息驱动、按需委派”的执行模式

2. 新 `multi_agent` 机制
- 由 `planner -> router -> executor` 图驱动
- 后端以 `task_pool + verified_facts + execution_state + final_result` 管理任务状态
- 前端已具备统一任务模型、Store 和基础 hydration 能力
- 本质是“工作流驱动、状态驱动、可恢复的多智能体编排”

本次方案升级的目标不是用新 `multi_agent` 替换 DeerFlow 原生 `leader agent`，而是明确二者的职责边界，并把它们收敛成一套可解释、可切换、可持续演进的双模式架构：

1. 保持原生 `leader agent` 链路不回归。
2. 让 `multi_agent workflow` 成为结构化任务的标准执行模式。
3. 在运行时引入 `auto / leader / workflow` 三档编排入口。
4. 前端继续只维护一套统一任务模型与状态展示体系。
5. 解决刷新恢复、重连恢复、多轮任务串台、等待澄清状态不可见、模式选择不透明等问题。

---

## 2. 总体原则

### 2.1 架构原则

1. 双模式共存，不二选一
- `leader` 适合开放式、探索式、工具驱动型任务
- `workflow` 适合结构化、跨领域、可恢复、可审计任务

2. 分层而非替代
- 上层决定“本轮使用哪种编排模式”
- 下层继续复用现有能力
- `workflow` 的 `executor` 继续复用 `make_lead_agent(...)` 作为领域执行单元

3. 前端单展示模型收口
- 不要求后端统一成同一协议
- 只要求前端将两种来源的任务归一化为统一 `TaskViewModel`

4. 事实源分层
- `legacy_subagent` 的恢复主源仍然是消息历史
- `multi_agent` 的恢复主源必须是 `thread.values.task_pool`
- `custom event` 只负责增量更新，不作为唯一恢复来源

5. 模式决策稳定
- 每一轮用户执行只解析一次编排模式
- 一旦进入某轮 `workflow`，本轮不再中途切回 `leader`
- 一旦进入某轮 `leader`，本轮不再补建伪 `task_pool`

6. 兼容优先
- 不破坏既有 DeerFlow 消息交互体验
- `workflow` 继续使用独立任务面板，不强依赖消息分组

### 2.2 模式定位原则

1. `Leader Mode`
- 语义：一个主代理主导全局，必要时调用 subagent
- 关键词：`open-ended`、`exploratory`、`tool-centric`

2. `Workflow Mode`
- 语义：显式 `planner -> router -> executor` 编排图驱动
- 关键词：`structured`、`multi-domain`、`recoverable`、`traceable`

3. `Auto Mode`
- 语义：入口先做一层轻量决策，再路由到 `leader` 或 `workflow`
- 关键词：`default`、`smart dispatch`、`progressive rollout`

### 2.3 非目标

本阶段不做以下事项：

1. 不统一 `subagent` 和 `multi_agent` 的后端执行引擎。
2. 不把 `multi_agent` 重新包装成伪 `task` tool call。
3. 不重写 DeerFlow 原有消息渲染体系。
4. 不在本阶段引入任务取消、任务重试、并行任务 DAG 可视化。
5. 不在本阶段做多级工作流嵌套。
6. 不在本阶段做基于历史统计的复杂模式学习。

## 4. 目标架构

改造完成后的目标结构如下：

### 4.1 总体架构

系统收敛为三档入口：

1. `auto`
- 默认入口
- 先做轻量决策，再选择进入 `leader` 或 `workflow`

2. `leader`
- 直接进入 DeerFlow 原生 `lead_agent`
- 适合开放式和探索式任务

3. `workflow`
- 直接进入 `planner -> router -> executor`
- 适合结构化和多领域协作任务

### 4.2 后端目标结构

1. `leader` 分支
- 保持现有 `make_lead_agent(...)` 为主入口
- 保持 `task` 工具与 `legacy_subagent` 链路
- 继续写消息历史，继续兼容旧前端交互

2. `workflow` 分支
- 保持现有 `planner -> router -> executor` 图
- 通过 `task_pool` 维护全量任务快照
- 通过 `verified_facts` 维护共享事实黑板
- 通过 `custom event` 提供增量更新
- 为每轮执行提供稳定 `run_id`

3. 分层关系
- `workflow` 不重造领域执行器
- `executor` 继续复用 `make_lead_agent(...)`
- 领域 agent 运行时仍是“受约束的 leader agent”

### 4.3 前端目标结构

1. 统一任务数据模型 `TaskViewModel`
2. 统一任务 Store `TaskStore`
3. 双来源归一化适配器
- `legacy_subagent -> TaskViewModel`
- `multi_agent -> TaskViewModel`

4. 模式感知展示
- `leader` 任务继续以消息嵌入式为主
- `workflow` 任务继续以 `TaskPanel` 为主
- `auto` 只展示本轮解析后的最终模式，不展示“切换过程”

### 4.4 模式选择矩阵

优先使用 `leader`：

1. 用户问题开放，目标尚未收敛。
2. 更像“先探索再决定下一步”。
3. 主要依赖技能、工具、网页、代码、文件操作。
4. 单领域即可完成，或者不需要显式共享事实。
5. 不需要前端任务面板、恢复点、审计轨迹。

优先使用 `workflow`：

1. 任务可明确拆解为多个子任务。
2. 明显跨领域，需要多个 domain agent 协作。
3. 需要共享中间事实，避免重复检查。
4. 需要澄清恢复、失败恢复、执行追踪。
5. 需要前端展示任务状态、等待澄清、完成汇总。

一句话区分：

- `leader` 适合“让 agent 自己想怎么做”
- `workflow` 适合“系统已经知道要按什么骨架来编排”

## 5. 模式选择与分流规则

### 5.1 目标

建立稳定、可解释、可灰度的模式选择机制，避免 `leader` 和 `workflow` 的职责不断漂移。

### 5.2 决策优先级

模式决策按以下优先级执行：

1. 用户显式指定
- 前端或 API 明确传入 `orchestration_mode=leader`
- 前端或 API 明确传入 `orchestration_mode=workflow`

2. Agent/助手级默认配置
- 特定 assistant 默认强制 `workflow`
- 特定 assistant 默认强制 `leader`

3. 运行时 `auto` 决策
- 未显式指定时才触发

### 5.3 `auto` 决策规则

建议采用“两段式决策”：

1. 第一段：确定性规则优先
- 命中强规则时直接决策
- 避免每轮都调用 LLM 做模式判断

2. 第二段：模糊场景轻量分类
- 仅在确定性规则不足时触发
- 可由轻量 prompt 或小模型做一次二分类

3. 第三段：兜底回退
- 分类失败时默认回退 `leader`
- 原因：`leader` 风险更低、兼容面更广

### 5.4 建议的强规则

直接进入 `workflow` 的强信号：

1. 用户目标中存在两个及以上并列子目标。
2. 明显需要两个及以上领域 agent 协作。
3. 任务需要等待澄清后继续执行。
4. 任务需要前端持续展示结构化进度。
5. 任务需要共享事实黑板、汇总验证、最终综合结论。
6. 任务预计较长，刷新恢复价值明显。

直接进入 `leader` 的强信号：

1. 明显是单步问答或单领域任务。
2. 目标尚不清晰，需要先探索再收敛。
3. 主要是代码、网页、文件、工具链操作，不需要显式编排。
4. 用户没有任何进度展示、恢复、审计诉求。

### 5.5 建议的评分规则

若需要更可控的 `auto`，可使用简单评分制：

```text
workflow_score = 0

+2: 多个并列子目标
+2: 需要多个 domain agent
+1: 需要等待澄清
+1: 需要刷新恢复
+1: 需要结构化任务面板
+1: 需要最终汇总验证

leader_score = 0

+2: 单领域
+2: 开放式探索
+1: 主要依赖通用工具链
+1: 用户仅需要最终答案，不关心过程
```

决策建议：

- `workflow_score >= 3` 且大于 `leader_score`：进入 `workflow`
- 否则进入 `leader`

### 5.6 模式决策结果持久化

建议将以下字段写入线程状态：

```python
requested_orchestration_mode: NotRequired[Literal["auto", "leader", "workflow"] | None]
resolved_orchestration_mode: NotRequired[Literal["leader", "workflow"] | None]
orchestration_reason: NotRequired[str | None]
```

要求：

1. `requested_orchestration_mode` 记录用户或前端请求。
2. `resolved_orchestration_mode` 记录本轮实际执行模式。
3. `orchestration_reason` 记录本轮分流原因，便于调试和前端解释。
4. 新轮次开始时重新计算 `resolved_orchestration_mode`。
5. 同一轮执行期间不允许切换 `resolved_orchestration_mode`。

## 10. 阶段 3：模式共存与入口分流

### 10.1 目标

把“原生 `leader agent`”与“新 `workflow`”从代码层的并存，升级为产品层、配置层、线程状态层的正式共存机制。

### 10.2 主要改动文件

后端建议新增或修改：

- `backend/src/agents/thread_state.py`
- 新增 `backend/src/agents/orchestration/selector.py`
- 新增 `backend/src/agents/orchestration/prompt.py`
- 新增 `backend/src/agents/entry_graph.py` 或等效入口工厂
- `backend/langgraph.json`
- `backend/src/gateway/routers/...` 中与 assistant / thread 启动参数相关的入口

前端建议新增或修改：

- `frontend/src/core/threads/types.ts`
- `frontend/src/core/threads/hooks.ts`
- 线程创建/发送消息相关调用层
- 聊天页面顶部状态栏或 header 组件

### 10.3 任务一：定义正式模式枚举

#### 目标

把临时性的“这个线程是不是 `multi_agent`”判断，升级为正式可传递、可持久化、可解释的编排模式。

#### 建议定义

前后端统一：

```ts
type RequestedOrchestrationMode = "auto" | "leader" | "workflow";
type ResolvedOrchestrationMode = "leader" | "workflow";
```

#### 实施要求

1. 前端发送消息时允许传入 `requested_orchestration_mode`。
2. 后端运行时解析得到 `resolved_orchestration_mode`。
3. 未指定时默认 `requested_orchestration_mode = "auto"`。
4. 对旧线程缺省值保持兼容，不要求补数据迁移。

#### 验收标准

1. 任意线程都能回答“用户请求的是哪种模式”和“系统实际执行的是哪种模式”。

### 10.4 任务二：引入模式选择器

#### 目标

为系统提供一个轻量的、可灰度的、可解释的 `auto` 决策器。

#### 建议实现

新增模式选择器模块，职责包括：

1. 读取用户输入、assistant 配置、thread 上下文。
2. 先执行确定性规则。
3. 必要时执行轻量分类。
4. 返回：
- `resolved_orchestration_mode`
- `orchestration_reason`
- 可选 `score` 与调试信息

#### 建议接口

```python
class OrchestrationDecision(TypedDict):
    requested_mode: Literal["auto", "leader", "workflow"]
    resolved_mode: Literal["leader", "workflow"]
    reason: str
    workflow_score: int
    leader_score: int
```

#### 验收标准

1. `auto` 模式可稳定输出 `leader` 或 `workflow`。
2. 决策结果可被日志、线程状态、前端 UI 读取。

### 10.5 任务三：新增统一入口图

#### 目标

避免“前端直接猜该调哪个 graph”，而是由后端统一入口负责模式分流。

#### 建议实现

建立统一入口：

```text
START
  -> orchestration_selector
    -> leader_entry
    -> workflow_entry
```

其中：

1. `leader_entry`

## 16. 关键风险与应对

### 风险 1：`run_id` 不落地，任务串台

影响：

- 多轮用户对话时，旧任务残留到新轮次

应对：

- 阶段 1 已完成 `run_id`
- 阶段 4 hydration 继续以 `run_id` 为第一过滤条件

### 风险 2：模式职责漂移

影响：

- 小任务被过度编排
- 大任务又落回 `leader`，失去恢复和可视化优势

应对：

- 阶段 3 必须先落地模式矩阵和 `auto` 规则
- `auto` 决策要可解释、可观测、可灰度

### 风险 3：消息流和任务面板重复展示

影响：

- 用户看到重复任务卡片

应对：

- 继续明确只让旧 `subagent` 在消息流展示
- `workflow` 只在 `TaskPanel` 展示任务态

### 风险 4：旧 subagent 体验回归

影响：

- 原本 DeerFlow 的 `task` 工具交互失效

应对：

- 旧消息分组逻辑保留
- 仅替换底层 Store 和卡片模型，不改其交互协议

### 风险 5：`auto` 决策不稳定

影响：

- 同类问题在不同轮次可能走不同模式，用户困惑

应对：

- 先用强规则，再用轻量分类
- 同一轮模式决策一旦确定即冻结
- 关键线程保留手动模式切换入口
