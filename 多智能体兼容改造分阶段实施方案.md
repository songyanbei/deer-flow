# 多智能体兼容改造分阶段实施方案

> 基于当前 `deer-flow` 代码现状，面向“原生 `leader agent` 模式与新 `multi_agent workflow` 模式共存、前端统一展示、运行时可自动分流”的落地实施文档。
>
> 文档版本：v1.4  
> 编写日期：2026-03-10  
> 当前状态：阶段 0-4 已落地，阶段 5 部分完成并持续收口，阶段 6 进入测试补齐与运行治理阶段；新增阶段 7 用于补齐串行协作式 workflow（求助 / 依赖 / 恢复）能力

---

## 1. 背景与目标

当前代码库中已经同时存在两套任务协作机制：

1. 原生 `leader agent` / 旧 `subagent` 机制
- 由 `lead_agent` 在对话过程中调用 `task` 工具触发
- 前端通过消息分组 `assistant:subagent` 渲染任务卡片
- 本质是“主代理驱动、消息驱动、按需委派”的执行模式

2. 新 `multi_agent` 机制
- 由 `planner -> router -> executor` 图驱动
- 后端以 `task_pool + verified_facts + execution_state + final_result` 管理任务状态
- 前端已具备统一任务模型、Store 和基础 hydration 能力
- 本质是“工作流驱动、状态驱动、可恢复的多智能体编排”

本次方案升级的目标不是用新 `multi_agent` 替换 DeerFlow 原生 `leader agent`，而是明确二者的职责边界，并把它们收敛成一套可解释、可切换、可持续演进的双模式架构：

1. 保持原生 `leader agent` 链路不回归。
2. 让 `multi_agent workflow` 成为结构化任务的标准执行模式。
3. 在运行时引入 `auto / leader / workflow` 三档编排入口。
4. 前端继续只维护一套统一任务模型与状态展示体系。
5. 解决刷新恢复、重连恢复、多轮任务串台、等待澄清状态不可见、模式选择不透明等问题。

---

## 2. 总体原则

### 2.1 架构原则

1. 双模式共存，不二选一
- `leader` 适合开放式、探索式、工具驱动型任务
- `workflow` 适合结构化、跨领域、可恢复、可审计任务

2. 分层而非替代
- 上层决定“本轮使用哪种编排模式”
- 下层继续复用现有能力
- `workflow` 的 `executor` 继续复用 `make_lead_agent(...)` 作为领域执行单元

3. 前端单展示模型收口
- 不要求后端统一成同一协议
- 只要求前端将两种来源的任务归一化为统一 `TaskViewModel`

4. 事实源分层
- `legacy_subagent` 的恢复主源仍然是消息历史
- `multi_agent` 的恢复主源必须是 `thread.values.task_pool`
- `custom event` 只负责增量更新，不作为唯一恢复来源

5. 模式决策稳定
- 每一轮用户执行只解析一次编排模式
- 一旦进入某轮 `workflow`，本轮不再中途切回 `leader`
- 一旦进入某轮 `leader`，本轮不再补建伪 `task_pool`

6. 兼容优先
- 不破坏既有 DeerFlow 消息交互体验
- `workflow` 继续使用独立任务面板，不强依赖消息分组

7. Workflow 展示单一承载
- `workflow` 模式下，实时子任务状态以 `TaskPanel` 为唯一主承载
- 主对话区只承载用户输入、顶层解释/总结、以及必要的澄清消息
- 不允许子图实时执行流在主对话区与 `TaskPanel` 双重渲染

### 2.2 模式定位原则

1. `Leader Mode`
- 语义：一个主代理主导全局，必要时调用 subagent
- 关键词：`open-ended`、`exploratory`、`tool-centric`

2. `Workflow Mode`
- 语义：显式 `planner -> router -> executor` 编排图驱动
- 关键词：`structured`、`multi-domain`、`recoverable`、`traceable`

3. `Auto Mode`
- 语义：入口先做一层轻量决策，再路由到 `leader` 或 `workflow`
- 关键词：`default`、`smart dispatch`、`progressive rollout`

### 2.3 非目标

本阶段不做以下事项：

1. 不统一 `subagent` 和 `multi_agent` 的后端执行引擎。
2. 不把 `multi_agent` 重新包装成伪 `task` tool call。
3. 不重写 DeerFlow 原有消息渲染体系。
4. 不在本阶段引入任务取消、任务重试、并行任务 DAG 可视化。
5. 不在阶段 0-6 做多级工作流嵌套；若后续引入 agent 间求助，优先在同一 `task_pool` 内实现串行依赖恢复，而不是引入嵌套子图。
6. 不在本阶段做基于历史统计的复杂模式学习。

5. LangGraph 本地 dev runtime 当前仅有单 worker；当已有长 workflow 占用 worker 时，新 workflow run 会停留在 `workflow_entry` 且 `task_pool` 为空，前端暂时没有显式的“排队中/准备中”占位卡片。
6. 某些 domain-heavy workflow 仍可能在 `planner validate -> 新任务` 之间反复往复，拉长单 run 占用时长，进一步放大前述队列等待问题。
7. `SubtaskCard` 已能承载 prompt、状态、结果、报错，但尚未形成稳定的结构化输入/输出展示区，也没有完整暴露模式来源标签。
8. 测试已覆盖 selector、entry graph、hydration、消息过滤和 workflow progress，但 `TaskPanel` 自身仍缺少专门的 UI 行为测试；灰度开关、埋点与治理指标也还未落地。

## 4. 目标架构

改造完成后的目标结构如下：

### 4.1 总体架构

系统收敛为三档入口：

1. `auto`
- 默认入口
- 先做轻量决策，再选择进入 `leader` 或 `workflow`

2. `leader`
- 直接进入 DeerFlow 原生 `lead_agent`
- 适合开放式和探索式任务

3. `workflow`
- 直接进入 `planner -> router -> executor`
- 适合结构化和多领域协作任务

### 4.2 后端目标结构

1. `leader` 分支
- 保持现有 `make_lead_agent(...)` 为主入口
- 保持 `task` 工具与 `legacy_subagent` 链路
- 继续写消息历史，继续兼容旧前端交互

2. `workflow` 分支
- 保持现有 `planner -> router -> executor` 图
- 通过 `task_pool` 维护全量任务快照
- 通过 `verified_facts` 维护共享事实黑板
- 通过 `custom event` 提供增量更新
- 为每轮执行提供稳定 `run_id`

3. 分层关系
- `workflow` 不重造领域执行器
- `executor` 继续复用 `make_lead_agent(...)`
- 领域 agent 运行时仍是“受约束的 leader agent”

### 4.3 前端目标结构

1. 统一任务数据模型 `TaskViewModel`
2. 统一任务 Store `TaskStore`
3. 双来源归一化适配器
- `legacy_subagent -> TaskViewModel`
- `multi_agent -> TaskViewModel`

4. 模式感知展示
- `leader` 任务继续以消息嵌入式为主
- `workflow` 任务继续以 `TaskPanel` 为主
- `auto` 只展示本轮解析后的最终模式，不展示“切换过程”

5. `workflow` 主页面布局原则
- `workflow` 执行期间，线程页面布局保持稳定，不因子任务执行切换到新的主视图
- `TaskPanel` 作为实时监控面板始终存在于当前线程页面中
- 规划、执行、等待澄清、汇总阶段都在同一面板容器内推进，而不是在消息区临时接管主视图

6. `workflow` 主对话区展示边界
- 保留用户真实输入
- 保留顶层 assistant 的模式说明、最终总结、必要的澄清消息
- 过滤 workflow 子图执行期间产生的中间 AI/tool 气泡、内部 task prompt、内部上下文注入消息
- 不让子任务实时执行流在主对话区制造“页面跳转感”或“中间结果气泡接管感”

7. `workflow` 面板内信息承载原则
- 子任务卡片负责承载该任务的输入、状态、增量进度、等待澄清、结果输出
- `TaskPanel` 负责展示本轮全局状态，`SubtaskCard` 负责展示单任务局部状态
- 面板中的实时状态更新以 `task_pool + custom event` 为主，不以消息分组协议为主

8. 交互体验复用原则
- 保留旧 `subagent` 子任务卡片的视觉和交互体验
- `SubtaskCard` 与统一任务状态映射继续作为共享组件能力沉淀
- 复用的是“卡片形态、状态文案、展开收起、进度/结果/澄清交互”
- 不复用旧 `assistant:subagent` 消息分组协议作为 `workflow` 的主承载方式

9. 输入入口分层
- 现有 `Flash / Reasoning / Pro / Ultra` 继续表示模型推理/执行风格
- 新增独立的编排方式入口：`Auto / Leader / Workflow`
- 编排方式入口在输入框区域与现有模式选择器并列呈现，而不是混入同一个下拉

### 4.4 模式选择矩阵

优先使用 `leader`：

1. 用户问题开放，目标尚未收敛。
2. 更像“先探索再决定下一步”。
3. 主要依赖技能、工具、网页、代码、文件操作。
4. 单领域即可完成，或者不需要显式共享事实。
5. 不需要前端任务面板、恢复点、审计轨迹。

优先使用 `workflow`：

1. 任务可明确拆解为多个子任务。
2. 明显跨领域，需要多个 domain agent 协作。
3. 需要共享中间事实，避免重复检查。
4. 需要澄清恢复、失败恢复、执行追踪。
5. 需要前端展示任务状态、等待澄清、完成汇总。

一句话区分：

- `leader` 适合“让 agent 自己想怎么做”
- `workflow` 适合“系统已经知道要按什么骨架来编排”

## 5. 模式选择与分流规则

### 5.1 目标

建立稳定、可解释、可灰度的模式选择机制，避免 `leader` 和 `workflow` 的职责不断漂移。

### 5.2 决策优先级

模式决策按以下优先级执行：

1. 用户显式指定
- 前端或 API 明确传入 `orchestration_mode=leader`
- 前端或 API 明确传入 `orchestration_mode=workflow`

2. Agent/助手级默认配置
- 特定 assistant 默认强制 `workflow`
- 特定 assistant 默认强制 `leader`

3. 运行时 `auto` 决策
- 未显式指定时才触发

### 5.3 `auto` 决策规则

当前代码使用“确定性优先 + 启发式打分”的轻量决策，不依赖额外 LLM 分类：

1. 第一段：确定性规则优先
- 用户显式指定 `leader / workflow` 时直接采用
- assistant 默认 `requested_orchestration_mode` 存在时直接采用
- 若当前输入被识别为澄清回复，则沿用上一轮 `resolved_orchestration_mode`

2. 第二段：启发式打分
- 识别多目标连接词、列表结构、workflow 关键词、leader 关键词
- 比较 `workflow_score` 与 `leader_score`

3. 第三段：兜底回退
- 未命中 workflow 阈值时回退 `leader`
- 原因：`leader` 风险更低、兼容面更广

### 5.4 建议的强规则

当前 selector 中直接影响 `workflow_score` 的主要信号：

1. 用户目标中存在两个及以上并列子目标。
2. 输入命中 workflow 关键词，例如 `workflow / report / research / plan / compare / validate / summarize / 并行 / 多步 / 调研 / 报告 / 总结 / 规划`。
3. 文本结构看起来像列表式、多步骤或多目标任务。

当前 selector 中直接影响 `leader_score` 的主要信号：

1. 输入较短，且不像多目标任务。
2. 命中 leader 关键词，例如 `explore / brainstorm / how to / what is / why / quick / search / browse / code / file / 网页 / 搜索 / 代码 / 文件`。
3. 场景更像开放式探索、单领域处理或工具驱动任务。

### 5.5 当前评分规则

当前 `backend/src/agents/orchestration/selector.py` 里的核心规则可概括为：

```text
workflow_score = 0

+2: 看起来像多个并列目标或列表任务
+0~2: 命中 workflow hints

leader_score = 0

+1: 输入较短
+1: 不像多目标任务
+0~2: 命中 leader hints
```

决策建议：

- `workflow_score >= 3` 且 `workflow_score > leader_score`：进入 `workflow`
- 否则进入 `leader`

### 5.6 模式决策结果持久化

建议将以下字段写入线程状态：

```python
requested_orchestration_mode: NotRequired[Literal["auto", "leader", "workflow"] | None]
resolved_orchestration_mode: NotRequired[Literal["leader", "workflow"] | None]
orchestration_reason: NotRequired[str | None]
```

要求：

1. `requested_orchestration_mode` 记录用户或前端请求。
2. `resolved_orchestration_mode` 记录本轮实际执行模式。
3. `orchestration_reason` 记录本轮分流原因，便于调试和前端解释。
4. 新轮次开始时重新计算 `resolved_orchestration_mode`。
5. 同一轮执行期间不允许切换 `resolved_orchestration_mode`。

class OrchestrationDecision(TypedDict):

    leader_score: int

#### 验收标准

#### 验收标准

## 16. 关键风险与应对

### 风险 1：`run_id` 不落地，任务串台

影响：

- 多轮用户对话时，旧任务残留到新轮次

应对：

- 阶段 1 已完成 `run_id`
- 阶段 4 hydration 继续以 `run_id` 为第一过滤条件

### 风险 2：模式职责漂移

影响：

- 小任务被过度编排
- 大任务又落回 `leader`，失去恢复和可视化优势

应对：

- 阶段 3 必须先落地模式矩阵和 `auto` 规则
- `auto` 决策要可解释、可观测、可灰度

### 风险 3：消息流和任务面板重复展示

影响：

- 用户看到重复任务卡片
- 用户误以为系统在子任务执行时“跳到了另一个页面”
- 主消息区被中间结果气泡临时接管，削弱 `workflow` 面板作为主监控界面的作用

应对：

- 继续明确只让旧 `subagent` 在消息流展示
- `workflow` 只在 `TaskPanel` 展示任务态
- `workflow` 主消息区增加显式过滤规则，屏蔽子图执行期的中间 AI/tool 气泡

### 风险 4：旧 subagent 体验回归

影响：

- 原本 DeerFlow 的 `task` 工具交互失效

应对：

- 旧消息分组逻辑保留
- 仅替换底层 Store 和卡片模型，不改其交互协议

### 风险 5：`auto` 决策不稳定

影响：

- 同类问题在不同轮次可能走不同模式，用户困惑

应对：

- 先用强规则，再用轻量分类
- 同一轮模式决策一旦确定即冻结
- 关键线程保留手动模式切换入口

### 风险 6：单 worker 队列等待导致 workflow 不可见

影响：

- 新 workflow run 在已有长任务占用 worker 时只能排队
- 排队期间线程状态可能停留在 `workflow_entry`，`task_pool` 为空
- 前端会暂时退化为通用 loading indicator，用户误以为 workflow 面板未加载

应对：

- 运行治理层面提高 LangGraph dev worker 并发或隔离长任务事件循环
- 前端在已确定 `workflow` 但尚未出现任务池时，补一个显式的“排队中/准备中”占位状态

### 风险 7：Planner validate 循环导致 worker 长时间占用

影响：

- 单个 workflow run 在 `validate -> 生成新任务 -> executor -> validate` 之间往复
- 任务虽然不是死循环报错，但会显著拉长 run 时长并挤压后续队列

应对：

- 增加 planner 收敛保护与场景化回归测试，重点覆盖 meeting/clarification 这类 domain-heavy workflow
- 将“长时间无总结收敛”纳入运行治理指标与灰度观察项

## 20. 阶段 7：串行协作式 Workflow 增强（新增，规划中）

### 20.1 背景

《多智能体架构技术路线说明书》明确提出了“轻量 Planner / Router + 持久化领域 Agent + 共享黑板”的目标形态，并要求领域 Agent 在超出能力边界时能够主动发起协助请求（yield），由路由层重新分派，形成自愈闭环。

当前代码已经具备以下基础：

1. `planner -> router -> executor` 的 workflow 编排骨架已经稳定落地。
2. `task_pool + verified_facts + execution_state + run_id` 已形成共享黑板的最小闭环。
3. `executor` 已支持 `ask_clarification` 面向用户的中断恢复。
4. 前后端已经能稳定展示 workflow 任务、进度和恢复状态。

当前缺口不在“是否有 workflow”，而在“workflow 是否支持 agent 间的串行协作”。当前实现仍然只支持：

1. Planner 拆任务。
2. Router 选 agent。
3. Executor 执行单个任务。
4. 缺信息时向用户澄清。

它还不支持以下链路：

1. domain agent 在执行中发现自己缺少外部能力或关键事实。
2. domain agent 发起结构化求助，请另一领域 agent 处理一个辅助任务。
3. helper task 完成后，把结果回写共享黑板。
4. 原父任务拿到依赖结果后恢复执行，而不是直接失败或回退到用户澄清。

### 20.2 本阶段目标

本阶段的目标不是一步到位实现技术路线说明书里的“并行调度 + 持久化领域 Agent + HITL 完整审批”目标态，而是在**保持当前串行执行模型不变**的前提下，先补齐下面这条闭环：

1. 复杂问题进入 `workflow`
2. 子任务路由给某个 domain agent
3. domain agent 发现缺少自己无法获取的事实，发起 `request_help`
4. 系统在同一 `task_pool` 中创建 helper task，并将父任务置为依赖等待
5. helper task 完成后，结果写回 `verified_facts`
6. 父任务自动解除阻塞并恢复执行
7. Planner 最终校验并输出结果

这意味着本阶段是**串行协作式 workflow**，而不是并行 workflow：

1. 同一时刻仍只允许一个任务进入实际执行。
2. helper task 与父任务按依赖关系串行推进。
3. 依赖恢复发生在同一个平铺 `task_pool` 中，不引入嵌套子图。

### 20.3 目标示例

以“预定会议室”演示链路为例：

1. 用户要求预定会议室。
2. Planner 生成会议预定任务。
3. Router 将任务分配给 `meeting-agent`。
4. `meeting-agent` 在执行中发现预定接口需要员工 `openid`，但该能力超出其工具边界。
5. `meeting-agent` 发起 `request_help(target_agent="attendance-agent", ...)`。
6. Executor 把当前会议任务标记为 `WAITING_DEPENDENCY`，并生成一个新的 helper task。
7. Router 将 helper task 分配给 `attendance-agent`。
8. `attendance-agent` 查询到 `openid`，结果写回 `verified_facts`。
9. 系统检测到会议任务依赖已满足，将其恢复为可执行状态。
10. Router 再次把会议任务交回 `meeting-agent`，并把已解析出的依赖结果拼入上下文。
11. `meeting-agent` 完成会议室预定。
12. Planner 做最终校验并向用户输出完成结果。

### 20.4 范围与边界

#### 本阶段纳入范围

1. 在现有 `task_pool` 内引入串行依赖关系，而不是新增嵌套图。
2. 新增 domain agent 间的结构化求助协议 `request_help`。
3. 支持父任务进入 `WAITING_DEPENDENCY` 并在 helper task 完成后恢复。
4. 支持把 helper task 的结构化结果回写到 `verified_facts` 和父任务上下文。
5. 前端在 `TaskPanel` / `SubtaskCard` 中可见“等待依赖 / 已恢复”状态。

#### 本阶段不纳入范围

1. 不做 `asyncio.gather` 并行调度。
2. 不做多级工作流嵌套或任意深度任务树。
3. 不做常驻进程级的持久化领域 Agent 实例。
4. 不在本阶段引入完整 HITL 审批流，只保留后续扩展接口。
5. 不引入复杂 DAG 可视化。

### 20.5 协议与状态改造

#### 20.5.1 `task_pool` 状态扩展

当前 `TaskStatus.status` 只有：

1. `PENDING`
2. `RUNNING`
3. `DONE`
4. `FAILED`

本阶段建议扩展为：

1. `PENDING`
2. `RUNNING`
3. `WAITING_DEPENDENCY`
4. `DONE`
5. `FAILED`

其中：

1. `WAITING_DEPENDENCY` 表示父任务已执行到一半，但需要等待 helper task 提供依赖结果。
2. 状态机仍保持单向推进，禁止 `DONE/FAILED` 回退。
3. `WAITING_DEPENDENCY -> PENDING/RUNNING` 的恢复只能由系统检测到依赖满足后触发。

#### 20.5.2 `TaskStatus` 字段建议新增

建议在 `backend/src/agents/thread_state.py` 中补充以下字段：

1. `parent_task_id`
2. `depends_on_task_ids`
3. `requested_by_agent`
4. `help_request`
5. `resolved_inputs`
6. `blocked_reason`
7. `resume_count`

建议语义如下：

1. `parent_task_id`
   helper task 的父任务 ID；普通任务为空。
2. `depends_on_task_ids`
   父任务当前依赖的 helper task 列表。
3. `requested_by_agent`
   谁发起了求助，便于前端展示和调试。
4. `help_request`
   结构化的求助内容，例如目标 agent、需要的结果、原因。
5. `resolved_inputs`
   已经从依赖任务中解析并可重新注入给父任务的结构化输入。
6. `blocked_reason`
   当前为什么进入 `WAITING_DEPENDENCY`。
7. `resume_count`
   父任务被恢复执行的次数，用于防止异常循环。

#### 20.5.3 `verified_facts` 结构增强

当前 `verified_facts` 已能按 `task_id` 写入结构化摘要；本阶段建议继续增强，但保持向后兼容：

1. `agent`
2. `task`
3. `summary`
4. `payload`
5. `fact_type`
6. `source_task_id`
7. `updated_at`

其中：

1. `summary` 继续用于 Planner 汇总和前端摘要展示。
2. `payload` 用于把 helper task 的关键结构化结果回写给父任务，例如 `openid`。
3. `fact_type` 与 `source_task_id` 用于后续扩展更丰富的依赖匹配和调试能力。

### 20.6 后端实现方案

#### 20.6.1 新增结构化求助协议 `request_help`

建议新增一个内置工具或等价的结构化协议，推荐命名为：

1. `request_help`

其参数建议至少包括：

1. `target_agent`
2. `task_description`
3. `reason`
4. `expected_output`
5. `merge_hint`

约束如下：

1. 仅 domain agent 在 workflow 执行中可调用。
2. 它不直接执行 helper agent，而是把“求助意图”结构化返回给 `executor`。
3. 它与 `ask_clarification` 并列，分别表示“向其他 agent 求助”和“向用户澄清”两种不同中断原因。

建议改动文件：

1. `backend/src/tools/builtins/` 新增 `request_help` 工具
2. `backend/src/agents/lead_agent/prompt.py` 为 domain agent 增加何时使用 `request_help` 的明确规则
3. `backend/src/tools/tools.py` 控制该工具只在 workflow/domain-agent 场景暴露

#### 20.6.2 `executor` 升级为“串行协作协调器”

`executor` 仍保持单任务执行入口，但在收到 `request_help` 时，不再把这看作普通失败，而是执行以下动作：

1. 保留父任务描述与当前执行上下文。
2. 将父任务状态改为 `WAITING_DEPENDENCY`。
3. 生成一个新的 helper task，状态为 `PENDING`。
4. 通过 `parent_task_id / depends_on_task_ids / help_request` 关联父子任务。
5. 发出新的 custom event，例如：
   - `task_waiting_dependency`
   - `task_help_requested`

helper task 完成后，`executor` 或 `router` 应负责：

1. 将 helper task 结果写入 `verified_facts`
2. 检查是否满足父任务的全部依赖
3. 若满足，则把父任务恢复为 `PENDING`
4. 把 helper task 的关键结果提取到父任务的 `resolved_inputs`
5. 发出 `task_resumed` 事件

这样可以在不引入嵌套图的情况下，完成“发起求助 -> 处理依赖 -> 恢复父任务”的闭环。

#### 20.6.3 `router` 扩展为“可恢复任务选择器”

当前 `router` 只做：

1. 选第一个 `PENDING` 任务
2. 分配领域 agent

本阶段应扩展为：

1. 优先选择无依赖的 `PENDING` 任务
2. 跳过 `WAITING_DEPENDENCY` 任务
3. 当 helper task 完成后，识别其父任务是否可恢复
4. 将满足依赖的父任务重新推进回 `PENDING`
5. 维持当前串行模型，一次只放行一个可执行任务

这样 `router` 仍是轻量节点，但开始承担“从黑板中挑选下一个可运行任务”的职责。

#### 20.6.4 `planner` 保持轻量，只做最小适配

本阶段不建议把复杂依赖编排逻辑继续堆到 `planner` 中。`planner` 仍只负责：

1. 首轮拆解
2. 所有可执行任务完成后的 validate
3. 新轮次识别和重置

建议补充的仅有：

1. validate 时把 `WAITING_DEPENDENCY` 视为未完成，而不是可忽略状态
2. 汇总 `verified_facts` 时兼容增强后的结构化字段
3. 避免在 helper task 尚未完成时误判为“无新任务可做”

### 20.7 前端展示改造

本阶段前端不需要大改布局，重点是补齐状态语义：

### 20.9 验收标准

本阶段完成后，应满足以下结果：

1. domain agent 可以在 workflow 执行中向其他 domain agent 发起结构化求助，而不是只能向用户澄清。
2. helper task 与父任务共享同一个 `task_pool` / `run_id`，不会引入新的嵌套图。
3. helper task 完成后，父任务可以自动恢复执行。
4. 刷新页面或重连后，`WAITING_DEPENDENCY` / 恢复中的任务状态可从 `task_pool` 恢复。
5. 当前串行执行模型保持不变，不因为引入求助而提前进入并行调度。
6. 对现有 `leader` 链路与 `ask_clarification` 链路无回归。

### 20.10 工作量评估

基于当前代码基础，若保持“串行协作，不做并行，不做持久化实例”，建议工作量按以下估算：

1. 后端状态协议与 `executor/router` 改造：4-6 天
2. `request_help` 工具 / prompt / 结构化回写：2-3 天
3. 前端任务状态与面板展示补齐：1-2 天
4. 回归测试与联调：2-3 天

总计约：

1. 单人：8-12 个工作日
2. 两人并行（后端 / 前端与测试分工）：5-7 个工作日
