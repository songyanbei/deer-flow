# 多智能体架构技术路线说明书

## 一、我们要解决什么问题

### 1.1 核心痛点：大规模 MCP 工具集下的选择退化

在实际企业业务中，我们积累了 **50+ MCP 工具**，覆盖多个业务域。当我们将所有工具一次性注入单个 LLM 上下文时，出现了严重的**工具选择准确率退化**问题：

- **理论分析**：当候选工具数 N > 30，LLM 在工具描述相似度较高时的选择准确率从 ~90% 骤降至 ~50%。这是因为 LLM 的注意力机制在大量相似候选项中发生"注意力稀释"，无法稳定区分语义相近的工具。
- **实际表现**：在生产环境中，模型频繁调用错误的工具，例如将"查询员工考勤"路由到"查询会议日程"的 MCP 工具，导致业务流程中断。
- **根因**：这不是模型能力不足，而是**架构设计缺陷**——单一 Agent 承载过多工具，超出了 LLM 单次决策的可靠边界。

### 1.2 解决思路：领域隔离 + 两级路由

我们提出的核心解法是**领域 Agent 隔离**：

```
用户请求
  → Planner / Router（只认识 5~8 个领域 Agent 的描述）
    → 领域 Agent A（只持有本域 5~10 个工具）
    → 领域 Agent B（只持有本域 5~10 个工具）
    → ...
```

这种**两级选择架构**将工具选择问题分解为两步：
1. **第一级**：从 5~8 个领域 Agent 中选择正确的领域（准确率 ~90%）
2. **第二级**：在选中的领域内从 5~10 个工具中选择正确的工具（准确率 ~90%）
3. **综合准确率**：90% × 90% = **~81%**，远高于直接从 50+ 工具中选择的 ~50%

即使第一级选错了领域，领域 Agent 也能通过**能力边界感知**主动发起协助请求（yield），由路由层重新分派，形成自愈闭环。

### 1.3 企业级要求

除了解决工具选择问题，我们对最终方案还有以下企业级要求：

| 维度 | 要求 |
|------|------|
| **稳定性** | 生产级错误处理、超时保护、防死循环机制 |
| **可扩展性** | 新增业务域只需添加 Agent 配置，无需修改核心代码 |
| **可观测性** | 完整的执行链路追踪、状态可视化 |
| **安全性** | 工具执行前的人工审批（HITL）、敏感操作拦截 |
| **复杂任务处理** | 多步骤任务拆解、子任务并行执行、跨域协作 |
| **记忆管理** | 工作记忆 + 持久化记忆，支持长对话和上下文恢复 |
| **工程质量** | 完善的测试覆盖、结构化日志、CI/CD 集成 |

## 二、现有框架各自的问题

我们深入调研了三个候选框架：自研的 Laifu-Agent-Core、社区热门的 OpenClaw（原 OpenHands）、以及字节跳动开源的 DeerFlow。以下逐一分析其核心问题。

### 2.1 Laifu-Agent-Core：验证了思路，但工程成熟度不足

Laifu-Agent-Core 是我们团队用 3 天时间快速搭建的原型系统，核心代码量约 1500 行。它成功**验证了领域 Agent 隔离 + 语义路由的可行性**，但距离企业级仍有显著差距。

#### 2.1.1 架构亮点（值得保留的设计理念）

- **YAML 驱动的 Agent 注册**：通过声明式配置定义 Agent 的工具集、能力描述、LLM 参数，新增业务域零代码侵入。
- **语义路由器**：基于 LLM 的动态路由，支持快速通道（hint_agent）、子任务返回、防死循环保护（MAX_ROUTE_COUNT=12）。
- **共享黑板（Blackboard）架构**：`GlobalBlackboardState` 提供跨 Agent 的状态共享，支持 `task_stack`（调用栈）和 `verified_facts`（已验证事实池）。
- **HITL 三级判断**：元数据标记 → 描述解析 → 关键词匹配，逐级递进的敏感操作拦截。

#### 2.1.2 核心问题

**问题一：Sentinel 字符串黑客——脆弱的跨层通信**

系统通过工具返回值中的特殊字符串标记实现流程控制：

```python
# engines/tools.py
return f"__YIELD_TRIGGERED__:{missing_information_desc}"
return f"__ASK_HUMAN__:{question}"

# engines/base_engine.py - 解析端
parts = result.split("__YIELD_TRIGGERED__:")
```

这种"Sentinel 字符串"模式存在严重隐患：
- LLM 可能在输出中意外生成包含 sentinel 的文本，触发误判
- `str.split()` 解析对格式变化极其敏感，缺乏容错
- 多种 sentinel（YIELD / ASK_HUMAN / HITL）的解析顺序和优先级无显式保证
- 对比 LangGraph 原生的 `interrupt()` / `Command(goto=END)` 机制，sentinel 方案完全绕过了框架的中断管理能力

**问题二：无任务拆解能力**

当前系统没有 Planner 组件，无法将复杂用户请求拆解为可并行执行的子任务。路由器只做"当前应该交给谁"的单步决策，缺乏全局任务规划视角。

**问题三：串行执行，无并行能力**

`task_stack` 是一个严格的调用栈结构，所有子任务只能按 LIFO 顺序逐个执行。当一个复杂请求可以分解为 3 个独立子任务时，系统仍然串行处理，效率低下。

**问题四：无记忆系统**

缺少工作记忆管理（token 预算控制）和持久化记忆（跨会话知识复用）。长对话场景下，早期上下文会被截断丢失，影响决策质量。

**问题五：工程化短板**

| 维度 | 现状 | 差距 |
|------|------|------|
| 日志 | `print()` 直接输出 | 无结构化日志框架 |
| 测试 | 零测试覆盖 | 无单元测试、集成测试 |
| 错误处理 | 基础 try-catch | 无统一错误码、无重试机制 |
| 配置管理 | 硬编码 + 环境变量 | 无分环境配置、无配置校验 |
| 类型安全 | TypedDict 基础类型 | 缺少运行时校验 |

#### 2.1.3 结论

Laifu 的价值在于**验证了领域隔离架构的正确性**，其 YAML 注册、语义路由、黑板共享的设计理念应当保留。但代码本身距离生产级有 2~3 个月的迭代差距，且需要从零构建中间件、记忆、沙箱等基础设施。

### 2.2 OpenClaw：生态繁荣但架构理念不匹配

OpenClaw（前身 OpenHands）是当前社区最热门的多智能体平台，GitHub 80K+ Star，支持 Web/Desktop/移动端多平台。

#### 2.2.1 架构特征

- **技术栈**：TypeScript（前后端统一），Node.js 运行时
- **Agent 模型**：基于 Skill 的可插拔能力扩展，每个 Agent 是独立的运行时实例
- **协作方式**：`sessions_spawn` 异步子任务委派，非阻塞，结果通过 `announce` 自然语言文本返回
- **并发控制**：`maxChildrenPerAgent=5`，`maxConcurrent=8`，最深 5 级嵌套
- **社区提案**：Agent Teams RFC（2026.2.5 Draft）计划引入共享任务列表 + 跨 Agent 消息传递，但**尚未实现**

#### 2.2.2 核心问题

**问题一：Agent 间完全隔离，无共享状态**

OpenClaw 的 `sessions_spawn` 虽然支持 Agent 间的任务委派，但本质上是**启动一个全新的独立会话**。子 Agent 无法访问父 Agent 的上下文、工具执行历史或中间结果。协作结果只能通过 `announce` 以自然语言文本返回，存在信息丢失。

对比我们的需求：领域 Agent 之间需要共享黑板上的事实池、任务状态、用户上下文，以实现高效的跨域协作。OpenClaw 的隔离模型不支持这一点。

Agent Teams RFC 虽然提出了共享任务列表的概念，但截至 2026 年 3 月仍处于 Draft 状态，距离可用还有较长周期。

**问题二：严重的安全隐患**

OpenClaw 在 2025 年底至 2026 年初爆发了严重的安全危机：

| 来源 | 发现 |
|------|------|
| Microsoft Threat Intelligence | 披露多个 Critical CVE |
| Cisco Talos | 确认远程代码执行风险 |
| Sophos X-Ops | 发现 824+ 恶意 Skill 包（"ClawHavoc"行动） |
| Palo Alto Networks Unit 42 | 报告 42,665 个暴露实例 |

6 周内 8 个 Critical CVE 的频率在主流开源项目中极为罕见，反映出安全审计流程的系统性缺失。对于需要接入企业内部 MCP 工具（含敏感数据操作）的场景，这一风险不可接受。

**问题三：技术栈不匹配**

我们团队的技术栈以 Python 为主，现有的 MCP Server、LLM 集成、内部中间件均基于 Python 生态。采用 TypeScript 技术栈的 OpenClaw 意味着：
- 团队需要切换技术栈，学习成本高
- 无法复用现有的 Python 生态组件（LangChain、LangGraph、各类 MCP SDK）
- 二次开发和长期维护的人力成本显著增加

**问题四：核心维护者流失**

OpenClaw 的核心创建者已于 2025 年底加入 OpenAI，项目的长期技术方向和维护力度存在不确定性。虽然社区仍在活跃，但核心架构决策缺少稳定的技术领导者。

#### 2.2.3 结论

OpenClaw 的生态繁荣和多平台支持令人瞩目，但其 **Agent 隔离架构与我们的共享状态需求根本矛盾**，叠加安全隐患、技术栈差异和核心维护者流失，不适合作为我们的基础框架。

### 2.3 DeerFlow：基础设施优秀但编排哲学需要重塑

DeerFlow 是字节跳动于 2025 年初开源的深度研究框架，基于 LangGraph 构建，具备成熟的工程化基础设施。

#### 2.3.1 架构优势（可直接复用的基础设施）

**11 层中间件管线**——这是 DeerFlow 最有价值的工程资产：

| 层级 | 中间件 | 功能 |
|------|--------|------|
| 1 | ThreadData | 线程数据注入 |
| 2 | Uploads | 文件上传处理 |
| 3 | Sandbox | 代码沙箱（Local/Docker/K8s 三模式） |
| 4 | DanglingToolCall | 悬空工具调用检测 |
| 5 | Summarization | 自动摘要（token 预算管理） |
| 6 | TodoList | 任务清单追踪 |
| 7 | Memory | 三级记忆（工作记忆 → 自动摘要 → 归档召回） |
| 8 | ViewImage | 图像处理 |
| 9 | SubagentLimit | 子 Agent 数量限制 |
| 10 | Clarification | 澄清追问 |
| 11 | ArtifactPresentation | 产物展示 |

每个中间件提供 4 个标准 Hook：`before_agent`、`before_model`、`after_model`、`wrap_model_call`，设计高度解耦。

**三级记忆系统**：
- 工作记忆：活跃上下文，受 token 预算控制
- 自动摘要：对话过长时自动压缩历史
- 归档召回：`memory.json` 持久化，支持跨会话知识复用

**沙箱隔离**：
- Local 模式：开发调试用
- Docker 模式：标准生产部署
- K8s 模式：弹性伸缩，多租户隔离

**SharedState（ThreadState）**：
- LangGraph 原生的图内状态共享，所有节点可读写
- 类似 Laifu 的 Blackboard 概念，但实现更规范

#### 2.3.2 核心问题

**问题一："Master 全知"模式——不可扩展的编排哲学**

DeerFlow 的核心编排逻辑是：一个 **Lead Agent（主 Agent）** 理解全部业务逻辑，动态创建和销毁子 Agent 来执行具体任务。

这种模式在单一领域（如深度研究）下工作良好，但在多业务域场景下存在根本性缺陷：

```
Master Agent 需要掌握的知识量 = 域数量 × 每域规则数
例：10 个业务域 × 5~8 条业务规则 = 50~80 条业务规则
```

当 Master 需要同时理解 HR 请假流程、会议预定规则、通讯录权限策略、项目审批流程……**遗漏任何一条规则都会导致整个业务链路断裂**。这与我们遇到的"50+ 工具选择退化"是同构问题——只是从工具层面上升到了规则层面。

**问题二：动态子 Agent 无法沉淀领域知识**

DeerFlow 的子 Agent 是临时创建、用完即销的。这意味着：
- 每次请求都要重新构建子 Agent 的工具集和 system prompt
- 无法积累领域特定的执行经验和优化策略
- 无法实现"Agent 级别的记忆"——同一领域的上次执行经验无法传递

对比我们的需求：领域 Agent 应当是**持久化的**，拥有固定的工具集、稳定的 system prompt、可积累的领域记忆。

**问题三：缺少 HITL（Human-in-the-Loop）机制**

DeerFlow 目前没有内置的敏感操作审批流程。在企业场景下，删除员工记录、批量发送通知、修改权限等操作必须经过人工确认，这是合规性的刚性需求。

**问题四：强耦合于研究场景**

DeerFlow 的默认工具集（Web 搜索、爬虫、代码执行）和流程设计（研究计划 → 信息收集 → 报告生成）高度面向"深度研究"场景。虽然架构本身可扩展，但需要较多改造才能适配通用企业业务场景。

#### 2.3.3 结论

DeerFlow 拥有**我们最需要的基础设施**（中间件、记忆、沙箱、前端），但其编排哲学（Master 全知 + 临时子 Agent）与我们的需求（轻量 Router + 持久化领域 Agent）存在根本差异。需要**保留基础设施，重塑编排层**。

## 三、我们应该采取什么样的路线

### 3.1 推荐方案：DeerFlow 基础设施 + Laifu 编排哲学

经过对三个框架的深入分析，我们推荐的技术路线是：

> **以 DeerFlow 为基座，保留其成熟的基础设施层（中间件、记忆、沙箱、前端），替换其编排层为 Laifu 验证过的"轻量 Planner/Router + 持久化领域 Agent + 共享黑板"架构。**

### 3.2 架构设计

```
┌─────────────────────────────────────────────────┐
│                    用户请求                       │
└──────────────────────┬──────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────┐
│          Planner / Router（轻量级）               │
│  · 只知道 N 个领域 Agent 的描述（不知道具体工具）   │
│  · 任务拆解：复杂请求 → 多个子任务                 │
│  · 路由分派：子任务 → 对应领域 Agent               │
│  · 完成校验：汇总结果，判断目标是否达成             │
└──────┬──────┬──────┬──────┬─────────────────────┘
       │      │      │      │
       ▼      ▼      ▼      ▼
┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐
│ HR   │ │会议  │ │通讯录│ │项目  │  ← 持久化领域 Agent
│Agent │ │Agent │ │Agent │ │Agent │     YAML 声明式注册
│5工具 │ │8工具 │ │6工具 │ │7工具 │     固定工具集
└──┬───┘ └──┬───┘ └──┬───┘ └──┬───┘     领域记忆可积累
   │        │        │        │
   ▼        ▼        ▼        ▼
┌─────────────────────────────────────────────────┐
│              共享黑板（Blackboard）                │
│  · task_pool: 平铺任务池（支持并行领取/回写）       │
│  · verified_facts: 已验证事实池                   │
│  · 执行状态追踪 / HITL 审批状态                    │
└─────────────────────────────────────────────────┘
       ▲        ▲        ▲        ▲
       │        │        │        │
┌──────┴────────┴────────┴────────┴───────────────┐
│           DeerFlow 基础设施层（复用）              │
│  · 11 层中间件管线（记忆/摘要/沙箱/上传/...）      │
│  · 三级记忆系统                                   │
│  · Docker/K8s 沙箱隔离                            │
│  · MCP 工具集成                                   │
│  · Web 前端（对话 + 可视化）                       │
└─────────────────────────────────────────────────┘
```

### 3.3 改造路径

改造工作分为四个阶段：

#### 第一阶段：编排层替换（~8 天）

| 任务 | 说明 | 工作量 |
|------|------|--------|
| Agent 注册中心 | 实现 YAML 驱动的持久化 Agent 注册，启动时加载配置、连接 MCP Server | 3 天 |
| 语义路由器 | 移植 Laifu 的 `semantic_router.py`，适配 DeerFlow 的 ThreadState | 2 天 |
| Planner 节点 | 新增任务拆解节点，输出结构化子任务列表 | 2 天 |
| LangGraph 图重构 | 替换 DeerFlow 的单 Lead Agent 图为 Router → Executor 多节点图 | 1 天 |

#### 第二阶段：共享黑板 + 并行执行（~7 天）

| 任务 | 说明 | 工作量 |
|------|------|--------|
| 黑板状态设计 | 将 `task_stack` 升级为 `task_pool`（平铺结构），支持状态标记（PENDING/RUNNING/DONE） | 2 天 |
| 并行调度器 | 实现基于 `asyncio.gather` 的多 Agent 并行执行 | 3 天 |
| 结果汇总 | Planner 从 task_pool 读取所有子任务结果，判断是否需要追加任务 | 2 天 |

#### 第三阶段：HITL + 流程控制（~5 天）

| 任务 | 说明 | 工作量 |
|------|------|--------|
| HITL 中间件 | 利用 DeerFlow 的中间件 Hook 实现 `before_agent` 级别的敏感操作拦截 | 2 天 |
| LangGraph 原生中断 | 用 `interrupt()` / `Command` 替代 Laifu 的 sentinel 字符串方案 | 2 天 |
| 前端审批 UI | 在 DeerFlow 前端增加确认/拒绝交互组件 | 1 天 |

#### 第四阶段：工程化收尾（~7 天）

| 任务 | 说明 | 工作量 |
|------|------|--------|
| 集成测试 | 编排层 + 路由 + 多 Agent 协作的端到端测试 | 3 天 |
| 日志 & 可观测性 | 结构化日志接入、执行链路追踪可视化 | 2 天 |
| 部署 & 文档 | Docker Compose 编排、配置文档、操作手册 | 2 天 |

**总计：约 27 天**（单人全职）。如果团队 2 人并行，可压缩至 **15~18 天**。

## 四、为什么要采取这条路线

### 4.1 为什么不选 OpenClaw

| 决策因素 | 分析 |
|----------|------|
| **架构匹配度** | Agent 间完全隔离（sessions_spawn 无共享状态），与我们的共享黑板需求根本矛盾。Agent Teams RFC 尚处 Draft 阶段，不可依赖。 |
| **安全性** | 6 周 8 个 Critical CVE，824+ 恶意 Skill 包，Microsoft/Cisco/Sophos/Palo Alto 四大安全厂商发出警告。企业内部 MCP 工具接入此生态风险极高。 |
| **技术栈** | TypeScript 全栈，与团队 Python 技术栈不匹配，改造和维护成本高。 |
| **持续性** | 核心创建者已离开加入 OpenAI，长期技术方向不确定。 |

### 4.2 为什么不从零迭代 Laifu

| 决策因素 | 分析 |
|----------|------|
| **基础设施缺口** | 中间件管线、记忆系统、沙箱隔离、前端——全部需要从零构建，工作量远超编排层改造。 |
| **工程成熟度** | 3 天原型代码到企业级，需要补齐的不仅是功能，还有测试、日志、错误处理、类型安全等工程基础，保守估计需要 2~3 个月。 |
| **重复造轮子** | DeerFlow 已经解决了我们 60%+ 的基础设施需求，没有理由重新实现。 |

### 4.3 为什么选择改造 DeerFlow

**核心逻辑：最大化复用比 + 最小化改造量。**

```
DeerFlow 提供的（直接复用）：
  ✓ 11 层中间件管线（~3000 行成熟代码）
  ✓ 三级记忆系统
  ✓ Docker/K8s 沙箱
  ✓ MCP 工具集成
  ✓ Web 前端 + SSE 流式输出
  ✓ 结构化日志
  ✓ 测试框架

我们需要改造的（~27 天工作量）：
  ✗ Lead Agent 全知模式 → 轻量 Planner/Router
  ✗ 临时子 Agent → 持久化领域 Agent（YAML 注册）
  ✗ 无 HITL → 中间件级审批流程
  ✗ 串行执行 → 共享黑板 + 并行调度
```

改造的本质是：**替换 DeerFlow 的"大脑"（编排逻辑），保留它的"身体"（基础设施）**。

这条路线的优势在于：
1. **避免重复造轮子**：直接复用 DeerFlow 数千行经过验证的基础设施代码
2. **编排哲学升级**：用 Laifu 验证过的领域隔离架构替代"Master 全知"模式，从根本上解决工具选择退化问题
3. **技术栈一致**：DeerFlow 基于 Python + LangGraph，与团队技术栈完全匹配
4. **渐进式改造**：可以分阶段推进，每个阶段都有可验证的交付物，风险可控
5. **社区背书**：字节跳动背景 + 活跃社区，长期维护有保障

### 4.4 最终目标

改造完成后，我们将获得一个具备以下特征的系统：

- **精准的工具选择**：两级路由将 50+ 工具的选择准确率从 ~50% 提升至 ~81%
- **灵活的领域扩展**：新增业务域只需添加 YAML 配置 + MCP Server，无需修改核心代码
- **高效的并行执行**：独立子任务由不同领域 Agent 并行处理，显著缩短复杂任务的端到端耗时
- **安全的操作审批**：敏感操作必须经过人工确认，满足企业合规要求
- **可靠的记忆管理**：三级记忆系统确保长对话不丢失上下文，跨会话知识可复用
- **生产级的工程质量**：完善的中间件管线、结构化日志、测试覆盖、容器化部署
