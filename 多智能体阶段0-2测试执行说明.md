本文档面向当前仓库状态，覆盖阶段 0-2 已落地能力的测试方法，重点验证两件事：

1. 新增的 `multi_agent` 状态协议、前端统一任务模型和事件分流是否正常。

2. 依赖后端运行时的脚本型验证

### 4.1 前端单元测试

执行命令：

```powershell

```

当前测试文件：

- [`frontend/src/core/tasks/adapters.test.ts`](/e:/work/deer-flow/frontend/src/core/tasks/adapters.test.ts)

覆盖点：

- legacy task event 不会被识别为 `multi_agent`

## 6. 手工测试用例

### 6.1 服务启动冒烟

#### CASE-SMOKE-01：服务成功启动

步骤：

1. 执行 `start-full-dev.ps1`
2. 打开 `http://127.0.0.1:3000`
3. 确认 Gateway 和 LangGraph 端口可访问

预期结果：

- 3000、8001、2024 三个端口正常监听
- 前端首页可以打开
- `.dev-runtime/logs` 下生成 3 个日志文件

#### CASE-SMOKE-02：服务成功停止

步骤：

1. 执行 `stop-full-dev.ps1`
2. 再次检查 3000、8001、2024

预期结果：

- 三个端口不再监听
- `.dev-runtime/services.json` 被删除

### 6.2 原有 lead_agent 聊天流程回归

#### CASE-LEADER-01：默认聊天页仍走原有 lead_agent

步骤：

1. 打开默认聊天页
2. 发送一个简单问题，例如“帮我总结一下今天要做什么”
3. 观察响应

预期结果：

- 可以正常返回回答
- 页面没有出现 `multi_agent` 专用任务面板
- 消息流正常工作

验证点：

- `useThreadStream` 没有把默认聊天页误切到 `multi_agent`

#### CASE-LEADER-02：自定义 Agent 聊天不回归

步骤：

1. 打开某个已有自定义 Agent 的聊天页
2. 发送一条消息
3. 观察回答和标题变化

预期结果：

- 能正常由对应 `agent_name` 处理
- 页面路径、标题、通知逻辑正常

#### CASE-LEADER-03：新建 Agent bootstrap 流程不回归

步骤：

1. 打开新建 Agent 页面
2. 输入一个新名字
3. 进入 bootstrap 聊天
4. 完成 `setup_agent` 相关流程

预期结果：

- 名称校验正常
- bootstrap 会话正常开始
- agent 创建成功后可继续聊天

### 6.3 原有 legacy subagent 流程回归

#### CASE-LEGACY-01：复杂任务触发 subagent 卡片

前提：

- 使用支持 `task` 工具调用的配置
- 最好在 Pro/Ultra 或原本可触发 subagent 的模式下测试

步骤：

1. 在默认聊天页输入一个复杂任务，要求拆分多个子任务
2. 观察消息流

预期结果：

- 消息区仍按 `assistant:subagent` 分组展示
- `SubtaskCard` 正常出现
- 不会出现 `multi_agent` 任务混入 legacy 卡片

#### CASE-LEGACY-02：legacy subagent 实时进度正常

步骤：

1. 触发 legacy subagent
2. 观察任务运行中卡片
3. 等待完成或失败

预期结果：

- `task_running` 能更新卡片内容
- 完成后显示 completed
- 失败或超时后显示 failed

验证点：

- source-less `task_*` event 被正确归类到 `legacy_subagent`

#### CASE-LEGACY-03：刷新后 legacy 任务历史仍可恢复

步骤：

1. 触发一个 legacy subagent 任务
2. 等它结束
3. 刷新页面

预期结果：

- 基于消息历史，卡片仍能恢复
- 不会因为刷新后 task event 丢失而变成空白
