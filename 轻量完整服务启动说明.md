# 轻量完整服务启动说明

## 1. 为什么建议用这套脚本

相比根目录的 `make dev`，这套脚本做了三件更适合本机联调的事：

1. 不启动 Nginx
- 少一个常驻进程
- 少一层反向代理
- 停止时不需要处理额外的 nginx 进程

2. 前端不走 Turbopack
- `make dev` 默认前端是 `next dev --turbo`
- Turbopack 在大型项目和 Windows 环境下更容易出现高 CPU / 高内存占用
- 这里改成了普通 `next dev`

3. 用 PID 文件管理进程
- 启动时记录到 `.dev-runtime/services.json`
- 停止时按记录和端口清理
- 不依赖 `pkill` 这种更偏 Unix 的方式

## 2. 启动命令

先统一编辑仓库根目录的 `.env`，这是本地开发的单一配置文件。前端、后端、启动脚本都会从这里读取配置。

```dotenv
DEER_FLOW_HOME=E:/work/deer-flow/backend/.deer-flow
CONTACTS_MCP_ENTRY=E:/path/to/mcp-servers/contacts/src/index.js
MEETING_ASSISTANT_MCP_ENTRY=E:/path/to/mcp-servers/meeting-assistant/src/index.js
TIME_SERVER_MCP_ENTRY=E:/path/to/mcp-servers/time-server/src/index.js
NEXT_PUBLIC_BACKEND_BASE_URL=http://127.0.0.1:8001
NEXT_PUBLIC_LANGGRAPH_BASE_URL=http://127.0.0.1:2024
```

说明：

- `DEER_FLOW_HOME` 要指向仓库内的 `backend/.deer-flow`
- `CONTACTS_MCP_ENTRY`、`MEETING_ASSISTANT_MCP_ENTRY`、`TIME_SERVER_MCP_ENTRY` 分别对应通讯录、会议室、时间服务的 MCP 入口脚本
- `NEXT_PUBLIC_BACKEND_BASE_URL`、`NEXT_PUBLIC_LANGGRAPH_BASE_URL` 供前端直连本地服务使用
- 如果不配置这些变量，`workflow` 模式可能加载不到 `contacts-agent`、`meeting-agent` 等领域智能体，请求会直接走系统兜底

配置好 `.env` 后，再执行启动命令：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\start-full-dev.ps1
```

启动后服务地址：

- Frontend: `http://127.0.0.1:3000`
- Gateway: `http://127.0.0.1:8001`
- LangGraph: `http://127.0.0.1:2024`

## 3. 停止命令

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\stop-full-dev.ps1
```

## 4. 日志位置

日志目录：`.dev-runtime/logs`

- `frontend.log`
- `gateway.log`
- `langgraph.log`

## 5. 当前已知结论

当前多智能体服务可以通过这套脚本正常启动和停止。

另外，启动链路里我顺手修复了一个真实问题：

- `multi_agent` 的 graph factory 签名原先不符合 LangGraph Server 要求
- 这会导致 `langgraph dev` 启动时直接报错
- 现在已修复为服务端工厂和测试工厂分离：
  - 服务端：`build_multi_agent_graph(config)`
  - 测试：`build_multi_agent_graph_for_test(checkpointer=...)`
